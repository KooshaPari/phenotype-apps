## 7. Process notes — autonomous subagent pattern

### 7.1 What worked

- **V5 plan as the single planning artifact.** The consolidated V5 plan ([`plans/2026-06-15-CONSOLIDATED-DAG-V5.md`](plans/2026-06-15-CONSOLIDATED-DAG-V5.md)) gave a 20×6 = 120 task DAG with per-task acceptance criteria. That density is what made autonomous fan-out tractable: each subagent had a numbered task and a known success gate from the same source.
- **Wave-level success criteria pinned up front.** The "Success Criteria" block at the bottom of the V5 plan ([lines 149-158](plans/2026-06-15-CONSOLIDATED-DAG-V5.md)) is the contract W4 / W5 / W6 must verify. Keeping that out of the wave-task bodies avoided "definition drift" mid-execution.
- **ADRs co-located with the wave output.** Three of the five ADRs are explicitly named as W6 tasks (12, 13, 14) and the other two are implicit in the W4 / W5 plan. Naming the ADRs as deliverables in the plan (rather than as a post-hoc doc dump) kept the architecture story tied to the wave that produced it.
- **Wide × shallow wave shape.** 20 tasks per wave with shallow per-task scope let the agent fleet parallelize without elaborate cross-task coordination. Topological hazards (e.g. "tests before CI") stayed inside one wave.
- **Repository state summary as the up-front truth table.** The V5 plan's first table ([V5 lines 6-13](plans/2026-06-15-CONSOLIDATED-DAG-V5.md)) gave every subagent the same baseline view of which repo was at what state — `Tasken | CLI wired, 67 tests pass`, `AgilePlus | Compilation fixed, merged to main`, `dispatch-mcp | Compiled, needs protocol compliance + test coverage`, `thegent-dispatch | Cleaned up (main), archived`, `cheap-llm-mcp | DEPRECATED`. That reduced re-discovery work.

### 7.2 What didn't (or didn't yet)

- **ADR files were not present in the working tree at harvest time.** All five ADRs are cross-linked in §5 at their planned paths, but `docs/adr/2026-06-15/` did not exist when the harvest was generated. This is the gap that makes "PR id", "tier count", "AI-DD crutch count", "coverage %", and "branch names" `pending` throughout this document. Resolution: pull the ADR bodies from the W6 worklog / branch state once W6 task 18 lands, then re-emit this harvest.
- **Wave-level "all green" claims are not in the V5 plan.** The plan defines the gates; the verdicts belong in worklogs. This harvest should not be treated as a green-light artifact — it is a *scope* artifact.
- **Cross-wave dependencies on the V5 plan's DAG edges** (e.g. W4 task 17-19 libification into `pheno-cli-base` / `pheno-errors` consuming the W1-W3 surface) are not validated here. V5 §W6 task 10 ("Cross-repo duplication report") and task 11 ("Cross-repo libification opportunities document") are the proper home for that evidence.
- **No verification commands were captured in this harvest.** The V5 plan's success criteria are verifiable (e.g. `cargo check`, `cargo test --no-run`, `cargo clippy -- -D warnings`, coverage threshold checks), but the harvest did not capture the actual command transcripts. That belongs in the W6 worklogs.
