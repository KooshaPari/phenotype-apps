# ADR-051 ‚Äî Promote `pheno-mcp-router` from lib tier to SDK tier

**Status:** PROPOSED (W6-3)
**Date:** 2026-06-19
**Owner:** pheno-mcp-router circle
**Tier:** lib ‚Üí **SDK**

## Decision

Promote `pheno-mcp-router` from a Python `lib` (V15 ¬ß83.4) to an SDK-grade substrate that any MCP server (Rust, Python, Go, TypeScript) can be built on top of, replacing per-MCP-server boilerplate (allowlist + sanitize + size-limit + logging).

## Background

`pheno-mcp-router` lives at `pheno-mcp-router/src/pheno_mcp_router/` and provides:
- `McpRouter` dataclass (name, backend_url, sanitize_keys, response_keys, max_message_bytes, max_response_bytes, timeout_seconds)
- `add_tier(name, route)` / `add_tool(tier, fn)` fluent API
- `_sanitize_payload` (strips keys not in `sanitize_keys`)
- `_allowlist_response` (strips keys not in `response_keys`)
- `dispatch(tier, payload)` via httpx
- CLI: `pheno-mcp-router init <name>`

Currently consumed by `dispatch-mcp` (1 consumer). Promotion to SDK requires:

1. **2+ in-production consumers** (currently 1: dispatch-mcp)
2. **Stable public API** (semver, deprecation cycle for breaking changes)
3. **Polyglot ergonomics** ‚Äî Python is the primary, but Rust/Go/TS servers need a way to consume the allowlist/sanitize logic

## Promotion criteria checklist

- [ ] `dispatch-mcp` is the only consumer; needs 1+ more
- [ ] `phenotype-mcp-sdk-py`, `phenotype-mcp-sdk-go`, `phenotype-mcp-sdk-ts`, `phenotype-mcp-sdk-rs` (4 SDKs) all exist as separate repos; should be merged INTO `pheno-mcp-router` or have a shared substrate
- [ ] No Rust binding exists; `pheno-mcp-router` is Python-only
- [ ] `llms.txt` exists (54 lines) but not generated by `pheno-llms-txt`
- [ ] No semver contract; the API is whatever the latest commit says

## Action items

1. **Adopt in 1 more consumer** ‚Äî `phenotype-mcp-sdk-go` (Go MCP server) or `phenotype-mcp-sdk-ts` (TS MCP server)
2. **Cut a v0.2.0 with semver contract** ‚Äî promise not to break public API in 0.x, only additions
3. **Generate `llms.txt`** via `pheno-llms-txt` template
4. **Document the `dispatch()` contract** ‚Äî payload/response schema, error codes, retry semantics
5. **Add `pheno-mcp-router-cli` entry point** ‚Äî `pip install pheno-mcp-router` should give a `pheno-mcp-router` binary

## Acceptance criteria

- [ ] 2+ consumers in production (e.g. `dispatch-mcp` + `phenotype-mcp-sdk-go`)
- [ ] v0.2.0 released with semver contract
- [ ] `llms.txt` ‚â§ 200 lines, generated by `pheno-llms-txt`
- [ ] `pheno-mcp-router-cli` works: `pheno-mcp-router init my-server` scaffolds a new MCP server
- [ ] Public API documented in `docs.rs`-equivalent (e.g. `mkdocs-material` or `mkdocstrings`)
