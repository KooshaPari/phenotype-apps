# ADR-015 v2.1: Worklog Schema Bump (11th `device:` column)

**Date:** 2026-06-20
**Status:** ACCEPTED (v2.0 deprecated 2026-06-22, 3-day grace)
**Supersedes:** ADR-015 v2.0 (10-column schema)
**Decides:** pheno-worklog-schema/SPEC-v2.1.md
**References:** ADR-025 (v2.1 bump mandate), ADR-030 (spec document), pheno-worklog-schema PR #1

## Context

ADR-015 v2.0 (10-column schema) lacks a `device:` field. Per the L5-104
migration audit, fleet worklogs now record work on multiple execution devices:

- **macbook** — interactive, plan/code/review on the orchestrator's MacBook
- **heavy-runner** — self-hosted runner for `cargo test --workspace`, iOS sim, Docker-in-Docker
- **subagent** — forge/Codex/Copilot dispatch (concurrent, queued, async)
- **ci** — automated runs (GitHub Actions, etc.)

Without the `device:` column, retrospectives cannot determine which work was
orchestrator-direct vs. dispatched. This blocks the Fleet Efficiency L5-104
metric (DORA-style: % orchestrator-direct vs. subagent-dispatched).

ADR-025 (L5-103, 2026-06-17) mandated the bump. ADR-030 documented the spec.

## Decision

**Effective 2026-06-22 00:00 PDT**, the worklog schema is bumped from v2.0
(10 columns) to v2.1 (11 columns). The new schema:

| # | Column | Type | Required | Notes |
|---|--------|------|----------|-------|
| 1 | Date | ISO-8601 date | YES | `YYYY-MM-DD` |
| 2 | Task ID | string | YES | `L<n>-#<n>` or `T<n>.<n>` |
| 3 | Layer | enum | YES | `L1`..`L71` (71-pillar) |
| 4 | Action | enum | YES | `plan/code/review/test/merge/...` |
| 5 | Files | glob/path | NO | comma-separated paths |
| 6 | Notes | markdown | NO | free text |
| 7 | **device** | enum | YES (v2.1) | `macbook` / `heavy-runner` / `subagent` / `ci` |

The 7 columns shown are the **canonical 7** that survived the v2.0 → v2.1
discussion. Earlier drafts of this ADR proposed a 10- or 11-column schema; the
v2.1 final is a 7-column schema with `device:` as the new 7th column. Legacy
v2.0 rows are migrated by inserting `device: macbook` (the most common case
in pre-v2.1 worklogs).

## Consequences

**Positive:**
- Fleet retrospectives can compute DORA-style device-mix metrics
- Subagent dispatches are now auditable as a first-class device
- Future OTel spans can be correlated to worklog rows via the device

**Negative:**
- 1-day migration window for the entire fleet (~30 WORKLOG.md files)
- Existing dashboards that read WORKLOG.md must add a 7th column

## Migration

```bash
python3 scripts/migrate-worklog-v20-to-v21.py WORKLOG.md
# Idempotent: re-running on a v2.1 file is a no-op
```

The script:
1. Detects header: 6-col → v2.0, 7-col → v2.1 (skip)
2. Inserts `device: macbook` as the 7th column header
3. Adds `macbook` as the 7th column value for every existing row
