# Guardrail: C4 component dependency map drift detection (GUARD-2-01)

**Date:** 2026-06-22
**Owner:** architecture circle
**Pillar:** L5 (Architecture — C4 component map)
**Status:** Proposed

## Current state

The fleet has 47+ substrate crates wired together via Cargo workspace paths, but
no machine-checkable record of *which* crate depends on *which*. New contributors
add `path = "../pheno-X"` lines without understanding the layering, so a leaf
crate ends up importing a federated-service crate and pulling in tokio + axum +
the whole OTLP stack. There is no CI gate that catches "leaf crate imports
service crate" violations. ADR-049 drift-detector regenerates the C4 dot files
weekly, but only flags *removed* edges — it does not flag *new, out-of-layer*
edges.

## Proposed guardrail

A `pheno-c4-lint` policy file (TOML) that lists each layer's allowed
up-dependencies (e.g. `pheno-config` may only depend on `pheno-errors`,
`pheno-tracing`, `pheno-context`). A CI step (`cargo metadata --format-version
1 | jq ...`) walks every workspace member's deps and fails the PR if any edge
crosses an illegal layer boundary. Errors mention the offending crate pair and
the layer rule.

## Concrete implementation

`pheno-c4-lint.toml` (~25 LOC) at monorepo root:

```toml
[layer.lib]
allow_up = ["layer.errors", "layer.tracing"]
deny = ["layer.service", "layer.framework"]

[layer.sdk]
allow_up = ["layer.lib"]
deny = ["layer.service"]
```

`.github/workflows/c4-lint.yml` (~20 LOC):

```yaml
- uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683
- run: cargo metadata --format-version 1 > meta.json
- run: ./scripts/c4-lint.sh meta.json pheno-c4-lint.toml
```

`scripts/c4-lint.sh` (~40 LOC, bash + jq) — emits "pheno-config → pheno-rpc
violates layer.lib.deny" for each bad edge, exits 1 on any.
