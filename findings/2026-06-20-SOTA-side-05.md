# side-05 — rust-analyzer perf for large workspaces

**Date:** 2026-06-20
**Scope:** All 22 `pheno-*` Rust crates + `phenoShared`, `FocalPoint`, `HexaKit`, `ResilienceKit`, `AuthKit` Rust halves, `phenotype-journeys`.
**Tracker:** FLEET_DAG `side-05`.

## Background

The Phenotype monorepo has **~58 Rust crates** in scope (22 pheno-* + ~36 in app/framework SDKs). rust-analyzer 2025-06 release notes (`release-2025-06-04`) cut workspace-wide analysis time ~25% via parallel proc-macro server pool and sysroot cache reuse. rust-analyzer 2025-Q4 (~0.85 dev cycle) added "package-centric flycheck" — runs `cargo check` per changed crate in parallel rather than serially. Reported cold-load for 50-crate workspace on M-series MacBook: ~12s vs ~28s prior. Hot-reload on edit: ~600ms median, ~1.5s p99. The fleet has no `.rust-analyzer.json` checked into the monorepo root today.

## Verdict

**ADOPT** rust-analyzer 2025-Q4 release with `package-centric flycheck` + per-crate `cfg(test)` exclusion. **REJECT** monolithic workspace analyze (still works, but 2-3x slower on cold load). **DEFER** rust-analyzer LSP-side **O(log n)** symbol search upgrade (not yet announced).

## Why

- Cold-load drop (12s vs 28s) matters for `pheno-wtrees` worktree switching: each new worktree spins a fresh analyzer. Saving 16s × 5-10 worktrees/day = 80-160s/day per fleet engineer.
- `cfg(test)` exclusion removes ~30% of crates from the dep graph (most `pheno-*` test modules pull in mockall, proptest, rstest).
- `proc-macro server pool` removes the per-crate proc-macro recompile which currently accounts for ~40% of edit-latency on the M-series MacBook.

## Action items

1. Add `.rust-analyzer.json` to monorepo root: `{"cargo.buildScripts.enable": true, "cargo.features": "no-console", "check.invocationLocation": "root", "check.overrideCommand": null, "cfg.test": false, "procMacro.enable": true}`.
2. `pheno-flake` (per ADR-039) ships a `nix develop` shell with rust-analyzer 2025-Q4 pinned. CI runner image bumps too.
3. Per-crate `.rust-analyzer.json` overrides in `pheno-port-adapter` and `pheno-mcp-router` (the heaviest) set `cargo.allTargets = false` to skip example/bin analysis unless open.
4. Add `rust-analyzer` version gate to `pheno-ci-templates`: PR fails if `rustup component add rust-analyzer --version < 2025-Q4` mismatches.
5. Doc: extend `pheno-cargo-template/SPEC.md` with the `.rust-analyzer.json` rationale; link from `pheno-scaffold-kit` generator output.

### Numbers & timing

- **Cold-load**: ~12 s (2025-Q4 rust-analyzer) vs ~28 s (prior) on M-series MacBook, 50-crate workspace.
- **Hot-reload on edit**: ~600 ms median, ~1.5 s p99.
- **Per-engineer time saved**: ~80-160 s/day (5-10 worktree switches × 16 s/cold-load delta).
- **Memory**: ~1-2 GB RAM at idle for 50-crate workspace; `cfg(test)` exclusion drops ~30% of dep graph.
- **proc-macro server pool**: cuts edit-latency ~40% by parallelizing per-crate macro compilation.

### Risks

- **Package-centric flycheck** is opt-in; not the default — must set `rust-analyzer.cargo.buildScripts.enable = true` explicitly.
- **rust-analyzer 2025-Q4** still under dev cycle; pin to release tag `2025-10-20` once stable to avoid nightly breakage on CI.
- **M-series vs Intel**: NEON codegen differs; verify per-crate benchmarks on both lanes if fleet ships Intel CI runners later.
