# Guardrail: pheno-tracing wire-up assertion per substrate crate (GUARD-2-08)

**Date:** 2026-06-22
**Owner:** observability circle
**Pillar:** L57 (Observability — pheno-tracing wiring)
**Status:** Proposed

## Current state

The v24 T2 plan wires `pheno-tracing` (2-line dep + `tracing_init()` call)
into 8 substrate crates. After merge, no gate prevents a future crate from
omitting the wire-up — and an un-instrumented crate ships spans to nowhere
or (worse) silently drops them when the OTel collector expects a
`service.name` resource attribute. Per `findings/71-pillar-2026-06-20-weekly-2.md`,
8 of 8 pheno-* repos lack pheno-tracing wiring today; this gap will close
once but the regression vector remains.

## Proposed guardrail

A `pheno-tracing-doctor` binary (in the `pheno-tracing` crate, ~80 LOC)
checks three invariants for any substrate repo: (1) `pheno-tracing` is in
`[dependencies]` of `Cargo.toml`, (2) `tracing_init()` (or equivalent) is
called in `lib.rs`/`main.rs`, (3) `service.name` resource attribute is
set before any span is emitted. The doctor runs in `cargo-deny.yml` (one of
the 4-workflow baseline from v24 T1) and fails the build on any miss. The
output names the missing piece with a one-line fix recipe.

## Concrete implementation

`pheno-tracing/src/bin/doctor.rs` (~80 LOC):

```rust
fn main() -> Result<(), Error> {
    let cargo = std::fs::read_to_string("Cargo.toml")?;
    if !cargo.contains("pheno-tracing") { fail("missing pheno-tracing dep"); }
    let lib = std::fs::read_to_string("src/lib.rs")
        .or_else(|_| std::fs::read_to_string("src/main.rs"))?;
    if !lib.contains("tracing_init") { fail("missing tracing_init()"); }
    Ok(())
}
```

CI step (3 LOC added to `cargo-deny.yml`):

```yaml
- run: cargo run -p pheno-tracing --bin doctor
```

## Effort

**S** — small binary, simple rules. ~half-day on MacBook. Backward
compatible: opt-in per ADR-040 grace period (one refresh cycle, 2 weeks).
