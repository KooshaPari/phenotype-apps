## 1. What was built

In-process plugin hot-reload for the `phenotype-router` Go spike. A long-running
`routerd` daemon can add, replace, or remove plugins **without restarting**,
triggered by either:

1. **SIGHUP** (`kill -HUP <pid>`) — operator-driven, canonical Unix "reload".
2. **File watcher** — polling-based (stdlib only, no `fsnotify` dep). Detects
   content-hash changes to the config file and the `.so` plugin directory.

Both triggers funnel through a single debounced reload consumer, so a
double-fire (e.g., SIGHUP arriving while the watcher is mid-tick) produces
**one** reload, not two.

### Public API

```go
// router.go
type Plugin interface {
    Name() string
    Kind() PluginKind
    Init(cfg map[string]any) error
    Shutdown(ctx context.Context) error
}
type Loader interface { Load(ctx, spec) (Plugin, error) }
type Registry struct { /* thread-safe map */ }

func NewRouter(loader Loader) *Router
func (r *Router) SetConfigPath(path string)
func (r *Router) ConfigPath() string
func (r *Router) CfgDir() string
func (r *Router) Config() Config
func (r *Router) Registry() *Registry
func (r *Router) ApplyConfig(ctx, cfg) (ReloadResult, error)   // pure data path
func (r *Router) ReloadPlugins(ctx) (ReloadResult, error)      // reads disk

type ReloadResult struct {
    Added, Removed, Replaced, Unchanged []string
}

// hotreload.go
type HotReloadOptions struct {
    PollInterval time.Duration  // default 2s
    Debounce     time.Duration  // default 200ms
    Logger       Logger
    Signals      []os.Signal    // default [SIGHUP]
}
func NewHotReload(router, cfgPath, pluginDir, opts) *HotReload
func (h *HotReload) Start(ctx) error   // launches SIGHUP + watcher + reload-loop goroutines
func (h *HotReload) Stop() error       // signals goroutines to exit, waits for them
func (h *HotReload) Reload(ctx, src) (ReloadResult, error)
func (h *HotReload) Done() <-chan struct{}
```

### Reload diff algorithm (in `ApplyConfig`)

```
wanted = enabled plugins from new config
current = snapshot of registry

for name, spec in wanted:
    if name not in current:          load + init + add      → Added
    elif spec_hash differs:          load + init + replace  → Replaced
    else:                                                    → Unchanged

for name in current:
    if name not in wanted:           shutdown + remove      → Removed
```

`sync.RWMutex` on Registry; reload takes the write lock for the duration of the
add/remove sequence (rare event; bounded by plugin Init/Shutdown cost). Hot
path (route dispatch) takes RLock.

### Debounce + content-hash design

- **Debounce window (200ms default):** Rapid triggers within the window are
  coalesced. The `Reload()` call arms a `time.AfterFunc`; if it fires before
  the next trigger, the actual reload runs. Multiple triggers → one reload.
- **Content hash (SHA-256 over config bytes + sorted .so listing):**
  `mtime`-only watchers fail on atomic-rename deploys (the destination
  preserves mtime) and `touch` without content change (false positive).
  Hashing the bytes is more robust and the cost is trivial for a router
  config that fits in tens of KB.

## 3. Decisions

### D1: Polling watcher, not `fsnotify`

- **Why:** Spike scope. The user mentioned fsnotify as "if needed" — for a
  spike whose goal is to validate the *registry lifecycle*, polling is
  sufficient and keeps the dep graph empty.
- **Tradeoff:** Up to `PollInterval` (default 2s) reload latency on a
  config-only deploy without SIGHUP. Acceptable for a spike; production
  would either:
  - Reduce `PollInterval` to 100-500ms, or
  - Implement a `Watcher` interface and slot `fsnotify.Watcher` in behind it.
- **Drop-in path:** `Watcher` interface would expose `Events() <-chan Event`
  + `Close() error`. Polling impl lives in this drop; fsnotify impl lands
  in a follow-up without touching hotreload.go.

### D2: One `runReloadLoop` consumer, not two

Initial draft had `handleSignals` and `runWatcher` each spawn their own
reload consumer. Two consumers fighting over `reloadReqCh` would race the
diff phase and double-apply. Fixed by:
- `Start()` spawns **one** `runReloadLoop`, **one** `handleSignals`, **one**
  `runWatcher`.
- Both producers send to `reloadReqCh`; the single consumer reads.
- A `sync.WaitGroup` tracks all three; `Stop()` waits for all to drain
  before returning.

This guarantees serial reloads and eliminates the dual-consumer race.

### D3: Content hash, not mtime

Mtime-only watchers break on:
1. **Atomic-rename deploys** — `mv new.json router.json` preserves the
   destination's mtime, so the watcher sees no change. Content-hash sees
   the bytes differ → reload fires correctly.
2. **Touch without content change** — mtime changes, watcher fires,
   reload runs, no diff → wasted work. Content-hash sees identical
   bytes → no-op.

### D4: `signal.Reset` in test cleanup

`signal.Notify` for SIGHUP is **process-wide**. Without cleanup, a test
that registers SIGHUP would leave the test binary's signal handler
changed after the test passes. `t.Cleanup(func() { signal.Reset(syscall.SIGHUP) })`
restores default behavior for the rest of the test binary and for any
subprocess that inherits the test process.

### D5: Stub `Loader` in `cmd/routerd/main.go`

The spike's daemon uses a stub loader (`builtinLoader`) that produces a
trivial `stubPlugin` from the spec. A real router would call
`plugin.Open(spec.Path)` (Go stdlib `plugin` package — Linux/macOS only,
in-process `.so` loading) and resolve symbols by name.

The plugin lifecycle (Load → Init → Shutdown) is exercised end-to-end;
the only missing piece is the symbol resolution, which is a separate
concern (the `Loader` interface is exactly the seam where it would slot in).

## 6. What's NOT in this drop (follow-ups)

1. **`plugin.Open` integration.** The stub loader is intentionally trivial.
   A real `Loader` impl would use Go's `plugin` package. Cross-platform
   caveat: `plugin` only works on Linux/macOS, not Windows — production
   would need a build tag + alternate loader for Windows.
2. **`fsnotify` watcher.** Documented as the drop-in replacement (D1 above).
3. **HTTP route dispatch.** The router's `Registry()` is the dispatch seam
   but `cmd/routerd/main.go` only serves a `/_/plugins` introspection
   endpoint. Wiring real route → plugin dispatch is a separate task.
4. **Graceful drain on plugin Shutdown.** Currently `Shutdown(ctx)` is
   called with the same context as the reload; a production version would
   pass a per-reload `context.WithTimeout(5*time.Second)`.
5. **Metrics.** No `pheno-otel` instrumentation. Would slot into the
   `Logger.Printf` call sites in `runReloadLoop` (4 events: reload start,
   reload success, reload error, no-change).
