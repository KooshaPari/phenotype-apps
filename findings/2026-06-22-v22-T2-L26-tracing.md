## Summary

This track closes the L26 (Tracing quality) P1 pillar for v22 cycle 12. The
substrate gains two production-grade additions:

1. A **sampling-policy module** with three canonical strategies (always-on,
   always-off, parent-based with a `TraceIdRatioBased` inner) plus two opt-in
   strategies (rate-limit, tail-based) for non-default use cases. Sampling
   decisions are O(1) per span and stable across the call graph.
2. A **cardinality-cap middleware** that limits span-attribute cardinality
   to 100 unique values per attribute name, with first-N-wins semantics and
   an overflow marker. The cap is applied in-place to a span's attribute
   map and is `Send + Sync` via `Arc`.

Both additions are wired into the crate root (`lib.rs`) for ergonomic
imports. Substrate quality-bar verification (`pheno-framework-lint check`)
passes — the crate remains a Tier 1 `pheno-*-lib` per ADR-042B.

## Cardinality cap proof

The cap middleware enforces 100 unique values per attribute name. First-N-wins
semantics — the first `cap` distinct values are passed through verbatim, all
subsequent distinct values are replaced with the overflow marker. Already-seen
values are always passed through verbatim (idempotent path).

**Proof of behavior (from `cardinality_cap_keeps_values_under_threshold`):**

```
cap = 100
input: 50 distinct user_id values (user-000..user-049)
expected: 50 kept, 0 overflowed
actual:   50 kept, 0 overflowed  ✓
seen-set size: 50 (under the 100 cap)
```

**Proof of overflow path (from `cardinality_cap_replaces_overflow_with_marker`):**

```
cap = 5
input: 12 distinct request_id values (req-000..req-011)
expected: 5 kept, 7 overflowed
actual:   5 kept, 7 overflowed  ✓
seen-set size: 5 (at the cap; never exceeds)
```

**Proof of idempotent path (from `cardinality_cap_preserves_already_seen_values`):**

```
cap = 5
batch 1: 5 distinct values (a..e)            → 5 kept, 0 overflowed
batch 2: 7 NEW values    (f..l)              → 0 kept, 7 overflowed
batch 3: 3 originals (a/b/c) + 2 new (m/n)   → 3 kept, 2 overflowed  ✓
```

The cap is per-attribute-name (`cardinality_cap_is_per_attribute_name` bonus
test), not global — `name_a` reaching its cap does not affect `name_b`. The
`reset()` method clears the seen registry for long-lived processes
(`cardinality_cap_reset_clears_seen_registry` bonus test).

**OTLP integration:** The middleware is designed to sit between
`TracePort::submit()` and the OTLP exporter, wrapping the per-span
attribute map:

```text
  caller ── submit() ──▶ TracePort ──▶ CardinalityCap::process()
                                                │
                                                ▼
                                          (in-place cap)
                                                │
                                                ▼
                                            OTLP exporter
```

A future v23+ track can wire this into the `InMemoryAdapter` and OTLP
exporter as a port-level pre-processor; the v22-T2 deliverable is the
middleware itself (the substrate contract), not the wiring.
