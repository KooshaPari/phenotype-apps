## 1. Summary

Implements the L31 pillar by computing a per-CI-run cache hit-rate from
`actions/cache` log lines (`Cache hit for …` / `Cache miss for …`), persisting
the per-run metrics to a versioned JSONL history under `docs/cache-stats/`, and
visualising the trend on a zero-dependency inline-SVG dashboard deployed to
GitHub Pages by a chained workflow.

The dashboard surfaces four views:

1. **Fleet KPI** — last-7-day weighted hit-rate, total runs, hit/miss counts.
2. **Per-repo table** — current hit-rate, runs, hits, misses (sorted by hit-rate asc to surface regressions).
3. **Daily trend** — line+area chart of per-day fleet hit-rate (last 14 days).
4. **Recent runs scatter** — one dot per run, x-axis = age in days, y-axis = hit-rate, colour = repo.

Empty-state is handled gracefully: when `history.jsonl` is empty (first deploy),
the dashboard shows a "no data yet" message and explains what to expect after
the first CI run.

### 2.1 `scripts/cache_stats_wrapper.sh` — MODIFIED (+137 / −51)

The v12 wrapper parsed a single log file and printed a JSON blob. This turn
extends it with three new modes needed for the workflow aggregator:

| Mode | Invocation | Purpose |
|---|---|---|
| (legacy) | `cache_stats_wrapper.sh <log> <repo>` | Single-file JSON blob (backward-compatible with v12 callers). |
| `--jsonl` | `cache_stats_wrapper.sh --jsonl <log> <repo> <run_id> <branch> <ts>` | Single-line JSONL record (used per-run by the workflow). |
| `--dir` | `cache_stats_wrapper.sh --dir <logs-dir> <fleet>` | Scan `<repo>-<run_id>.log` files and emit one JSONL record per file (used by the scheduled rollup). |
| `--aggregate` | `cache_stats_wrapper.sh --aggregate <history.jsonl>` | Compute the fleet rollup (weighted hit-rate, per-repo, per-day) — written to `fleet.json`. |

Repo name is extracted from the log filename via `^([^-]+(?:-[^-]+)*)-\d+\.log$`
(e.g. `pheno-tracing-12345.log` → repo=`pheno-tracing`, run_id=`12345`).

### 2.2 `.github/workflows/cache-stats.yml` — NEW (+191)

`workflow_run` trigger on the `CI` workflow (success/cancelled/failure) +
6-hour schedule + manual `workflow_dispatch`.

```yaml
on:
  workflow_run:
    workflows: ["CI"]
    types: [completed]
  schedule:
    - cron: "0 */6 * * *"
  workflow_dispatch: {}
```

Steps:

1. Download CI run logs via `gh run download --log-files` (or fall back to API).
2. Concatenate → run `cache_stats_wrapper.sh --jsonl` per repo → append to
   `docs/cache-stats/history.jsonl` (capped at 5000 records).
3. Compute `latest.json` from the most recent record and `fleet.json` from the
   last-7-day window.
4. **Bootstrap step** — on first run, copy `seed.jsonl` to `history.jsonl` so
   the dashboard renders meaningful data immediately after Pages deploy.
5. Commit and push (only if changed) using `ad-m/github-push-action` with
   identity `forge@phenotype.local`.

Permissions: `contents: write`, `actions: read` (per ADR-046 minimum).

## 5. What happens on the first CI run after merge

1. CI workflow completes on `main`.
2. `cache-stats.yml` fires (`workflow_run` event, type=`completed`).
3. `cache-stats.yml` downloads the CI run logs via `gh run download`.
4. The workflow parses `Cache hit` / `Cache miss` lines per repo.
5. JSONL record(s) appended to `docs/cache-stats/history.jsonl`.
6. `latest.json` + `fleet.json` regenerated.
7. `cache-stats.yml` commits back to the same branch via `ad-m/github-push-action`.
8. The commit lands on `main` → triggers `cache-stats-pages.yml` (which also
   listens for `push: paths: docs/cache-stats/**`).
9. `cache-stats-pages.yml` uploads `docs/cache-stats/` as a Pages artifact and
   deploys to `https://kooshapari.github.io/repos/cache-stats/`.

End-to-end latency: ~3-5 minutes from CI completion to dashboard refresh.

## 7. Risks & follow-ups

| Risk | Severity | Mitigation |
|---|---|---|
| GitHub Pages not enabled on `KooshaPari/repos` | M | Add a one-time "Pages: Build from branch `/`" config to repo settings; included in PR description. |
| `gh run download` requires `actions: read` + `contents: read` | L | Both are in the workflow's `permissions:` block. |
| Submodule pointer drift in this monorepo's local clone can race with `git push` of the history append | M | The `push` step in `cache-stats.yml` only commits `docs/cache-stats/*` (additions to existing files), so submodule noise is irrelevant to the remote. |
| `cache-stats.yml` fires on `pull_request` runs too | L | `if: github.event.workflow_run.event == 'push'` gate ensures only main-branch CI runs are recorded. |
| `history.jsonl` grows unbounded | L | Workflow caps at 5000 records (last-write-wins dequeue). At 1 record/run × 50 runs/day, that's a 100-day buffer. |
| L31 scorecard update | L | Re-run `findings/71-pillar-{date}.md` cycle on next Monday (per ADR-041 cadence). |
