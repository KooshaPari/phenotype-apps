## 2. Configuration recipe (`.cargo/config.toml`)

Three concerns, three tables. The file is opt-in — fleet repos copy it in.

```toml
[build]
rustflags = ["-C", "lto=thin"]                 # release-only LTO (no-op in dev)

[target.x86_64-unknown-linux-gnu]
linker = "lld"                                 # ~2-3x faster than ld on Linux

[profile.dev]
codegen-units = 256                            # max parallelism for clean/dev builds

[profile.release]
codegen-units = 1                              # max inlining for runtime
```

**Why this layout:** `-C lto=thin` is a no-op outside release, so it is safe in
`[build].rustflags`; per-profile `codegen-units` overrides any rustflag for
the same key and lives naturally in `[profile.*]`. Per-crate opt-out: set
`[profile.release] lto = false` in the crate's `Cargo.toml`.

## 4. sccache warm-up curve

Cumulative cache hits vs. build iteration (synthetic, fleet-typical):

```
iter │ hits │ misses │ hit_rate │ wall-clock_s
─────┼──────┼────────┼──────────┼─────────────
  1  │    0 │  412   │   0.00   │   420   (cold)
  2  │  395 │   17   │   0.96   │    45   (host-local hit)
  3  │  410 │    2   │   1.00   │     8
  4  │  411 │    1   │   1.00   │     7
  5  │  412 │    0   │   1.00   │     6
```

Steady-state ≥ 0.96 hit rate from iter 2 onward. Cross-host sharing is
sccache's S3/GCS backend; first pull from a peer adds ~5 s RTT, subsequent
pulls from the same artifact are local-cache fast.

**Local config**: `SCCACHE_DIR=$HOME/.cache/sccache` (default 10 GB).
`SCCACHE_CACHE_SIZE=10G` is the per-host ceiling. For CI, set
`SCCACHE_BUCKET=pheno-sccache-prod` + `SCCACHE_REGION=us-west-2` to enable
cross-host sharing.

## 6. Cross-references

### 6.1 ADR-040 (Test coverage gates per tier)

`docs/adr/2026-06-18/ADR-040-test-coverage-gates-per-tier.md`.

ADR-040 enforces 80/70/60% line coverage per substrate tier via
`pheno-coverage --min=$MIN`. **L35 complements L23 by reducing the
"cost of coverage"**: faster incremental builds let developers run
the full coverage suite on every save without the previous 45 s
penalty. T1 lib graduation (ADR-048 §1, see below) is now feasible
because the candidate crate can be built, tested, and coverage-gated
inside a single iteration.

### 6.2 ADR-048 (Substrate graduation path — T1 lib gate)

`docs/adr/2026-06-18/ADR-048-substrate-graduation-path.md` §1.

ADR-048 §1 lists the T1 (beta) graduation gates:
- ≥70 % test coverage
- CI on every PR
- CHANGELOG.md exists
- Has ≥1 consumer

**Build perf is the implicit 5th gate** (currently undocumented): a candidate
crate whose `cargo build --release` takes >10 min on CI blocks the
"iterate to coverage" workflow. With L35 in place, the gate becomes
"`cargo build --release` ≤ 5 min on a clean sccache-warmed CI runner" —
a soft, measurable check that `scripts/build-perf-track.py` already
instruments.

**Concrete action (proposed for v23):** add a `[build_perf]` table to
ADR-048 §1 with the L35 thresholds and reference this finding. The
existing `findings/build-perf-{date}.json` artifact is the natural
input to the gate.
