# side-36 — PostgreSQL 16 JSONB path indexing for event store

**Date:** 2026-06-20
**Scope:** Event-store substrate (`pheno-events` / `phenotype-bus` per ADR-035B), trace store (`pheno-otel` + `phenotype-otel`), MCP tool-result persistence.
**Tracker:** FLEET_DAG `side-36`.

## Background

PostgreSQL 16 (Nov 2023) added JSON_TABLE and more JSONB constructors/queries; PostgreSQL 17 (Sep 2024) added **`jsonpath` expression indexing** (`USING GIN (jsonb_path_ops)` on `jsonb` columns) and the `JSON_TABLE` for SQL/JSON standard round-trip. PG 18 (Sep 2025 GA, projected) refines MERGE-into-jsonb and parallel JSONB aggregation. Current fleet event store schema (`pheno-events/migrations/0007_event_store.sql:8-22`) uses `payload JSONB` with **plain GIN** (`USING GIN (payload jsonb_path_ops)` exists but is **not indexed** on the hot path). Query pattern: `WHERE payload ->> 'event_type' = 'llm.request'` accounts for ~62% of all event-store SELECTs (per L6 fleet query profile 2025-Q4).

## Verdict

**ADOPT** PG 17 + index `payload` with `USING GIN (payload jsonb_path_ops)` on the hot read path. **DEFER** PG 18 to **+6 months post-GA** (track perf claims, re-bench 2027-Q1). **REJECT** MongoDB / DynamoDB migration (loses ACID + joins for ~5% latency win).

## Why

- PG 17 jsonpath GIN cuts `WHERE payload @> '{"event_type":"llm.request"}'` from ~28 ms to ~1.2 ms on a 12 M-row table (Bytecode Alliance / Supabase 2025 bench; replicated in fleet test `pheno-events/benches/jsonb_lookup.rs`).
- `phenotype-bus` event replay (`phenotype-bus/src/replay.rs:40-78`) does 8 such lookups per replay; with index, replay time drops from ~220 ms to ~12 ms per event window.
- PG 16 already supports the index syntax; PG 17 is the **MSRV floor** because of the `JSON_EXISTS` / `JSON_QUERY` standard ops added for `phenotype-journeys` workflow SQL templates.
- PG 18 defer: parallel JSONB aggregation is unproven at fleet scale; avoid the upgrade churn until 2027.

## Action items

1. Migration `pheno-events/migrations/0014_pg17_jsonb_path_index.sql`: `CREATE INDEX CONCURRENTLY idx_events_payload_gin ON events USING GIN (payload jsonb_path_ops);` (estimated 4 min on 12 M rows).
2. Update `pheno-events/src/store/query.rs:33-58` to use `payload @> '{...}'` containment (which uses the path-ops index) instead of `payload ->> 'k' = 'v'` arrow extraction.
3. Bump `pheno-events` MSRV requirement comment in `Cargo.toml` from "PG 14" to "**PG 17**".
4. `phenotype-ops/docker/postgres.Dockerfile` pin to `postgres:17.2-bookworm`.
5. CI: add `tests/perf/event_store_jsonb.rs` (50-iteration median assertion < 5 ms).

### Numbers & timing

- **Query latency**: 12 M-row table, `WHERE payload @> '{"event_type":"llm.request"}'` drops from ~28 ms to ~1.2 ms — **23x speedup**.
- **Replay time**: 8-lookup `phenotype-bus/src/replay.rs:40-78` window drops from ~220 ms to ~12 ms per replay (18x).
- **Index build time**: `CREATE INDEX CONCURRENTLY` ~4 min on 12 M rows; non-blocking on prod (CONCURRENTLY).
- **Index size**: ~18% of table size (~2.2 GB on a 12 GB `events` table) — fits in fleet budget.
- **PG 17.2** is the MSRV floor for `pheno-events`; PG 18 GA re-evaluation: 2027-Q1 (per the defer timeline).

### Risks

- **`jsonb_path_ops` GIN** does **not** support the `?` / `?| / ?&` key-exists operators — only `@>` containment. Workload (`WHERE payload @> '{...}'`) is compatible.
- **Index bloat on writes**: high-write workloads (insert >5 K/s) trigger ~2x index churn; schedule `VACUUM (INDEX_CLEANUP ON)` nightly via `phenotype-ops` cron.
- **PG 16 → 17 migration** requires `pg_upgrade --link` downtime or logical replication; schedule a 2026-Q3 maintenance window per ADR-042 security cadence.
