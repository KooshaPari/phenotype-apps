# Guardrail: Latency p99 dashboard freshness (GUARD-2-02)

**Date:** 2026-06-22
**Owner:** observability circle
**Pillar:** L14 (Performance — latency p99 dashboard)
**Status:** Proposed

## Current state

Per `findings/2026-06-22-v24-L14-p99.md` (in-flight v24 T2), a Grafana panel
for per-route p99 latency is being authored for the 8 substrate services. Once
shipped, the dashboard will silently rot: a service that drops a probe (network
partition, OTel collector restart) reports a flat-line p99 of 0, and operators
treat 0 as "fast" rather than "missing." There is no freshness signal on the
panel itself, so a 3-day-outage shows up as "great news" to the on-call.

## Proposed guardrail

Every latency series pushed to the dashboard is tagged with a synthetic
`__probe_age_seconds` attribute set at scrape time. The Grafana panel includes
a secondary "probe freshness" stat panel that turns RED when any series is
older than 5 minutes (warn at 3 min, alert at 10 min via Alertmanager). The
freshness check is per-service, so a single dropped probe pages the right team.

## Concrete implementation

`pheno-otel` middleware (~15 LOC, attribute on every span):

```rust
span.set_attribute("__probe_age_seconds",
    (now() - last_successful_span_epoch).as_secs());
```

Grafana panel JSON (~30 LOC), threshold rules:

```yaml
- alert: ProbeStale
  expr: max(time() - __probe_age_seconds) by (service) > 300
  for: 2m
  annotations:
    summary: "{{ $labels.service }} probe stale >5m"
```

Wired into `phenotype-ops/grafana/alerts/probe-freshness.yaml` and gated by
ADR-041 weekly-cadence review.

## Effort

**S** — single attribute emission + 2 Grafana panels + 1 alert rule. ~half-day
on the MacBook. No schema migration; backward-compatible attribute.
