## 4. Port decisions

### 4.1 Same as toolrouter (V11 T2.5 port precedent)

| Concern | Source (Bifrost) | Target (Phenotype-Router) |
|---|---|---|
| Plugin interface | `schemas.Plugin` | `sdk.Plugin` (Bifrost v1.5 parity shape) |
| Request type | `*schemas.BifrostRequest` | `*sdk.Request` |
| Response type | `*schemas.BifrostResponse` | `*sdk.Response` |
| Context | `*context.Context` (pointer-to-context, source's pattern) | `context.Context` (stdlib idiom) |
| Embedding payload location | `req.EmbeddingRequest` (Bifrost union field) | `req.Embedding` (added in this port) |
| Error/short-circuit | `*schemas.BifrostError`, `*schemas.PluginShortCircuit` | `*sdk.Error`, `*sdk.ShortCircuit` |
| Provider name | `schemas.ModelProvider` (string type) | plain `string` |
| Secret handling | `config.APIKey` passed through | `config.APIKey` passed through, **but** `Config()` omits it (V12-T15 F2) |

### 4.2 Departures from toolrouter (V12-T07-specific)

| Departure | Why |
|---|---|
| **fasthttp + sonic → stdlib net/http + encoding/json** | toolrouter is pure routing logic and never opens a network connection. Voyage is a provider plugin that actually calls VoyageAI's HTTP API. The source's fasthttp + sonic stack trades two third-party deps for a small throughput win that does not matter at the per-router request rate of an embeddings API. Stdlib keeps the dep graph empty, matches every other plugin in this repo, and is what the broader Phenotype-Router spike standardizes on (per V12-T15 F4 — JSON parsing safety; V12-T15 F3 — TLS verify). |
| **pool.go (sync.Pool of fasthttp.Request/Response) deleted** | sync.Pool of fasthttp.Request/Response only exists because fasthttp.Request is a heavy struct (~24 fields). stdlib `http.Request` is a thin wrapper around the underlying conn; pooling it is not idiomatic and `httptest`-style servers (used by our tests) do not benefit from it. The 39-LOC file becomes zero. |
| **`makeBifrostError` → `sdk.ShortCircuit{Err: ...}`** | The source's `BifrostError` carries `StatusCode`, `Message`, and `Err *ErrorField`. The Phenotype-Router SDK's `ShortCircuit` carries `Reason` only (per V12-T15 F7: "Error message sanitization — never echo request bodies / analysis detail"). Status codes were never consumed by the router anyway, so this is a strict reduction. |
| **`schemas.EmbeddingRequest` → `sdk.EmbeddingRequest` (SDK addition)** | The Bifrost union type is fundamental to the source plugin's PreHook branching (`if req.EmbeddingRequest == nil`). The Phenotype-Router SDK had no embedding type until this port. We added `sdk.EmbeddingRequest`, `sdk.EmbeddingResponse`, `sdk.EmbeddingData` and two `*Embedding` fields (one on `Request`, one on `Response`). The additions are **purely additive** — chat-only plugins (`toolrouter`, `contentsafety`) ignore the new fields. Existing `toolrouter` and `contentsafety` test suites pass unchanged after the SDK additions. |
| **Retries actually implemented** | The source's `Config.MaxRetries` field is documented but the source plugin ignores it (the source's `p.client.Do` is called once per request with no retry loop). The port implements the documented retry semantics: exponential backoff (100ms floor, 2× multiplier), 0–50ms jitter, retry on 429 + 5xx, no retry on other 4xx. This is a strict improvement — the existing API contract (Config.MaxRetries) is honored. |
| **Pre-flight URL validation (M5 spirit)** | The source emits a confusing fasthttp error if `BaseURL` is malformed. The port validates `BaseURL` at `NewClient` time (must be absolute http/https, non-empty host) and rejects non-absolute schemes with `ErrEmptyBaseURL` at request time. This is the same defensive posture V12-T15 recommends for `phenotype-router`'s `cliproxy_base` URL. |
| **`apiKey` redacted from `Config()`** | The source has no `Config()` method (Bifrost dumps config internally). The Phenotype-Router SDK requires `Config()` for OTel resource attributes. The port explicitly omits `APIKey` from the returned map to prevent the secret from being emitted into spans/logs (V12-T15 F2 forward-looking flag). |

### 4.3 What was deliberately NOT ported

| Source | Decision | Why |
|---|---|---|
| The full `EmbeddingRequest.ExtraParams` passthrough (input_type, encoding_format, etc.) | Truncated to `input_type`, `output_dtype` for embedding; rerank handles `top_k`, `return_documents`, `truncation` only | The source's `ExtraParams` map carries any string→any and the plugin forwards whatever the caller sets. The port supports the documented VoyageAI-specific keys and ignores others. A follow-up can add a free-form `map[string]json.RawMessage` if more passthrough is needed; for now, the explicit fields cover the documented VoyageAI API surface. |
| `makeBifrostError`'s `StatusCode` field | Dropped | Per V12-T15 F7 — the SDK's `ShortCircuit` carries `Reason` only, and the router never reads status codes. |
| The source's TODO in `trackToolUsage` | Not applicable | The voyage plugin has no equivalent. |

## 8. Open questions / future work

1. **`ExtraParams` free-form passthrough** — the current port supports `input_type` and `output_dtype` for embedding; future VoyageAI keys (`encoding_format`, `embedding_dtype`, etc.) would be added as explicit fields. If the user community wants a free-form passthrough, that's a one-line addition (`for k, v := range req.ExtraParams { upstream.<key> = v }` for known keys).
2. **`Provider` constant in SDK** — the source uses `schemas.ModelProvider` (a typed string). The port uses plain `string` because the SDK's `Request.Provider` is already a plain string. If/when the SDK grows typed provider constants (e.g. `sdk.ProviderVoyage`), this plugin would switch.
3. **OpenAI-compatible embeddings adapter** — the SDK additions make it trivial to port the OpenAI embeddings plugin (when one is extracted from argis-extensions). The SDK's `Request.Embedding` / `Response.Embedding` are now the canonical shape; a future V12-T? ticket could add `internal/plugins/openai/` with the same skeleton.
