# Guardrail: Required CI workflow matrix per substrate repo (GUARD-2-07)

**Date:** 2026-06-22
**Owner:** pheno-ci-templates circle
**Pillar:** L29 (DX — CI matrix)
**Status:** Proposed

## Current state

Per `findings/71-pillar-2026-06-20-weekly-2.md`, 4 of 8 substrate repos
(`pheno-config`, `pheno-context`, `pheno-events`, `pheno-flags`) have no CI
workflows at all. The v24 T1 plan adds 4 workflows per repo (ci, deny, audit,
governance-drift), but nothing prevents a 9th substrate repo from being
created tomorrow without any CI. The `phenotype-registry` refresh (ADR-043)
flags a missing workflow per refresh, but only at bi-weekly cadence — by then
the repo has shipped 30 PRs on raw `cargo build` runs.

## Proposed guardrail

`pheno-ci-templates` ships a `validate-repo.yml` workflow that any
substrate repo can opt into via a single label in `.github/CODEOWNERS`. The
validator checks for the presence of `ci.yml`, `cargo-deny.yml`,
`cargo-audit.yml`, `governance-drift.yml` (the 4-workflow baseline from
v24 T1) and fails PRs that remove any of them. New substrate repos are
required to opt in during `pheno-registry` row creation (registry validation
script refuses to add a `lib`/`sdk`/`framework` row without a
`has_ci_matrix: true` flag, set automatically by the validator's first run).

## Concrete implementation

`.github/workflows/validate-repo.yml` (~40 LOC):

```yaml
- name: Check required workflows
  run: |
    for w in ci cargo-deny cargo-audit governance-drift; do
      [ -f .github/workflows/${w}.yml ] || \
        (echo "missing ${w}.yml"; exit 1)
    done
```

`phenotype-registry/scripts/add-row.py` (~15 LOC patch):

```python
if row["kind"] in {"lib", "sdk", "framework"}:
    assert row["has_ci_matrix"], "substrate must have CI matrix"
```

## Effort

**S** — single workflow + registry validation patch. ~half-day on MacBook;
adoption is opt-in per repo per ADR-040.
