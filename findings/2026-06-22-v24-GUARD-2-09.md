# Guardrail: Branch-protection ruleset drift detector (GUARD-2-09)

**Date:** 2026-06-22
**Owner:** governance circle
**Pillar:** L46 (Security — branch protection)
**Status:** Proposed

## Current state

Per `findings/71-pillar-2026-06-20-weekly-2.md`, 0 of 8 substrate repos have
branch protection rulesets configured. The v24 T5 plan creates rulesets for
the 8 pilot repos via `gh api` (heavy-runner). After merge, an admin can
silently disable protection via the GitHub UI (Settings → Branches → uncheck
"Require pull request reviews before merging") and no signal fires. A repo
can lose its protection and ship a force-pushed force-merge before anyone
notices.

## Proposed guardrail

A scheduled workflow (weekly Monday 09:00 PDT, per ADR-041) runs
`pheno-ruleset-audit` against all repos in the `KooshaPari/*` org that have
the `substrate` topic. The audit queries `gh api /repos/{owner}/{repo}/rules`
for each, compares against the canonical ruleset JSON in
`phenotype-ops/governance/branch-protection.json`, and opens a GitHub issue
tagged `security-drift` when any rule is missing or weaker. Severity
auto-escalates if the missing rule is `required_signatures` or
`required_linear_history`.

## Concrete implementation

`.github/workflows/ruleset-audit.yml` (~30 LOC, scheduled cron):

```yaml
on:
  schedule: [{ cron: "0 16 * * 1" }]   # Mon 09:00 PDT
jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - run: ./scripts/ruleset-audit.sh KooshaPari substrate
```

`scripts/ruleset-audit.sh` (~50 LOC, bash + `gh api jq`):

```bash
for repo in $(list_repos); do
  actual=$(gh api repos/KooshaPari/$repo/rules/branches/main)
  diff <(echo "$actual") canonical.json || \
    gh issue create --repo $repo --label security-drift --body "..."
done
```

## Effort

**S** — script + workflow. ~half-day on MacBook. Backward compatible:
informational issues for the first cycle, hard-fail alerts in cycle 2.
