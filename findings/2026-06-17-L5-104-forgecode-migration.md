**Why does DM92 exist at all?** Most likely a previous automated fork (e.g., from a "fork all of upstream's branches" GitHub App or a one-time `gh repo fork` with a post-fork `git push origin --all` from the upstream clone). The fork was created on 2026-06-15 and never received any post-fork commits — the `pushed_at` (2026-06-15T05:35:50Z) is 13 seconds after the `created_at` (2026-06-15T05:35:37Z), which is consistent with a single bulk-push event.

**Phenotype-relevance of the 364 mirror branches:** All of these are upstream feature branches (`feat/`, `fix/`, `chore/`, `feature/`, `refactor/`, `exp/`, `ext/`, `poc/`, `perf/`, plus unscoped names like `accept-or-reject-command-exec`, `acp-integration`, `atomic-model-provider`). **If a specific upstream feature is desired for Phenotype:** the right move is **not** to migrate from DM92, but to **cherry-pick the upstream commit directly into KP/forgecode** from `tailcallhq/forgecode` (or merge the upstream PR once it's been vetted). Going through DM92 would add an unnecessary hop and risk SHA confusion.

### 3.4 DM92 stale-mirror evidence (3 corroborating signals)
