## Policy cross-references

The release-train policy is intentionally layered on top of three
existing ADRs. The cross-references are not cosmetic — each ADR
imposes a constraint the train policy MUST honor:

### ADR-015 v2.1 (worklog schema, device column)

- File: [`docs/adr/2026-06-20/ADR-015-v2.1-worklog-schema.md`](../adr/2026-06-20/ADR-015-v2.1-worklog-schema.md)
- **Binding constraint:** Every worklog row in a release-train cycle
  MUST include the `device:` column (v2.1 schema; v2.0 deprecated
  2026-06-22).
- **Where honored:** `release-train.md:88-101` (owner ack chain); each
  calendar entry has a `device:` field.
- **Where it bites:** A W2 GA release cannot be logged from a MacBook;
  it MUST record `device: heavy-runner` per ADR-023 Rule 1.

### ADR-023 (agent-effort governance, device-fit gate)

- File: [`docs/adr/2026-06-15/ADR-023-agent-effort-governance.md`](../adr/2026-06-15/ADR-023-agent-effort-governance.md)
- **Binding constraint:** Rule 1 (device-fit gate) — heavy work
  (`cargo test --workspace`, 1k-RPS soak, full SBOM build, cosign
  signing) MUST run on a heavy runner or CI, not a MacBook.
- **Where honored:** `release-train.md:98-101`; calendar entries W2-W4
  carry `device: heavy-runner`; W1, W5, W6 carry `device: macbook`.
- **Where it bites:** A release captain who tries to run W2 GA on a
  MacBook triggers an L5 governance failure (ADR-023 Rule 1 violation).

### ADR-048 (substrate graduation, 4-tier gate table)

- File: [`docs/adr/2026-06-18/ADR-048-substrate-graduation-path.md`](../adr/2026-06-18/ADR-048-substrate-graduation-path.md)
- **Binding constraint:** Tier T0/T1 promotion is by substrate
  maintainer; T2 promotion requires 3-of-5 architect vote; T3
  requires fleet-owners unanimous.
- **Where honored:** `release-train.md:81-84` (tier priority in
  conflict resolution); calendar entries list per-week
  `substrate_in_focus` whose tier determines priority.
- **Where it bites:** A T0 alpha crate cannot ship to production in
  the W2 GA without first graduating to at least T1.

## Edge case table

| Scenario               | Detection                       | Policy response                                                                                          | Authority          |
|------------------------|---------------------------------|----------------------------------------------------------------------------------------------------------|--------------------|
| **Broken release**     | CI red on W2-Tue publish; or post-publish canary fails within 24h | Revert tag; open postmortem within 7 days; re-cut to W2 of next train (not this train's W3); SKIP slot NOT consumed (release-window failure is not a SKIP) | RC + Architect Circle |
| **Revert**             | Critical CVE-7+ or data-loss discovered post-GA | YANK via cosign (containers) or `cargo yank` (crates); revert to prior GA tag; counts as 1 SKIP against cycle cap (sec path) | Security Lead + RC |
| **Hotfix**             | Non-sec bug filed in W3-W4 against just-released GA | W3-W4 hotfix window: patch from released tag → publish as `x.y.z+1` within 24h; counts as 1 SKIP against cycle cap | RC |
| **Sec audit slip**     | Monthly `cargo audit` + `pip-audit` (ADR-042) misses W4 gate | Defer the SKIP if non-sev-1; do NOT slip W2 release date (sec audit slip ≠ release slip); add to next cycle's W1 hardening if sev-2/3 | Security Lead |
