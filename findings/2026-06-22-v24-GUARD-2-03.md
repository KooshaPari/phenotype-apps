# Guardrail: Cache hit-rate floor + regression detector (GUARD-2-03)

**Date:** 2026-06-22
**Owner:** performance circle
**Pillar:** L17 (Performance — cache tuning)
**Status:** Proposed

## Current state

Substrate crates (`pheno-config`, `pheno-context`) cache parsed TOML/JSON in
memory with no exposed hit-rate metric. The 2026-06-22 v24 T3 bench plan
targets hit-rate ≥ 80% in a 1k-RPS 30-min soak, but there is no runtime
assertion. After merge, a code change that accidentally bypasses the cache
(e.g. a new config-key path that re-parses every call) ships to production
unnoticed — the bench passes in CI because the bench fixture happens to hit
the cached path.

## Proposed guardrail

`pheno-cache` substrate crate exposes a `CacheMetrics` struct with
`hits`, `misses`, `evictions` atomic counters, scraped by `pheno-otel` every
10s. CI bench job asserts hit-rate ≥ 80% over a synthetic 1k-RPS replay of
production traces (replay fixture in `tests/fixtures/cache-replay.jsonl`).
Runtime: hit-rate < 50% for 5 min triggers a `CacheDegraded` warning
(not page, to avoid flapping on cold-start).

## Concrete implementation

`pheno-cache/src/metrics.rs` (~50 LOC):

```rust
pub struct CacheMetrics {
    pub hits: AtomicU64,
    pub misses: AtomicU64,
    pub evictions: AtomicU64,
}
impl CacheMetrics {
    pub fn hit_rate(&self) -> f64 { /* hits / (hits+misses) */ }
}
```

`.github/workflows/cache-bench.yml` (~30 LOC, reuses bench harness from v24 T3):

```yaml
- run: cargo bench --bench cache_hit_rate -- --warmup 5 --measurement 30
- run: ./scripts/assert-hitrate.sh 0.80
```

OTel emission via `pheno-tracing` macro on cache miss path (1 LOC per site).

## Effort

**S** for the substrate (`pheno-cache` already exists in skeleton form). **M**
for the bench + CI gate. Total: ~2 days on heavy-runner.
