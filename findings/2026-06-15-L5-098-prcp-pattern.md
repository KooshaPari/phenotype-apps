# Finding — PRCP (Prometheus Remote Compliance Protocol) pattern (L5-098)

**Date:** 2026-06-15
**Task:** L5-098
**ADR:** ADR-018
**Worklog:** `worklogs/L5-098-prcp-pattern-2026-06-15.json`

## Headline

ADR-018 ratifies the **PRCP (Prometheus Remote Compliance Protocol)**
pattern for non-Prometheus backends to consume the same wire format
as Prometheus remote_write, but emit compliance/security signals
instead of metrics. The pattern reuses the existing `prometheus-remote-write`
crate family with a 3-trait façade:

- `RemoteWriteSink` (counter/gauge/histogram) — 1:1 Prometheus semantics
- `ComplianceSink` (policy violations + audit events) — ADR-018
  semantic
- `UnifiedSink` (both, with a `route_by_label` config) — the
  default for the PhenoAgent daemon

## Why

- 5 pheno-* services (PhenoAgent, PhenoCompose, PhenoRuntime,
  PhenoData, PhenoMCP) were each building their own audit/compliance
  shipper with slightly different shapes (~200-500 LOC each).
- 2 of the 5 had unrelated bugs in the `Bearer` header handling
  that wouldn't have shipped if they shared the wire format with
  Prometheus.
- The PRCP pattern is a 50-LOC addition to `pheno-tracing` that
  unifies all 5 with a single wire format + automatic schema
  validation via `jsonschema` (built into the `prometheus` crate
  family since 0.13).

## Implementation

- `pheno-tracing/src/sink/prcp.rs` (50 LOC): thin wrapper around
  `prometheus::protobuf::remote::WriteRequest` with the compliance
  field extension
- `pheno-tracing/src/sink/mod.rs`: trait re-exports
- 5 consumer services migrate to `prcp::UnifiedSink` over the next
  3 sprints (one per sprint, tracked in ADR-018 follow-up section)

## Verification
