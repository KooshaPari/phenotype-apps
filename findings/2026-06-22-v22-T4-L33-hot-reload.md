## 2. Signal-handler design

### 2.1 Problem statement

Long-running pheno-* daemons (MCP-router, observability sidecar, event bus, registry cache) need to pick up non-secret config changes (feature flags, log levels, rate-limit thresholds) without a process restart. The fleet-standard signal for "reload config" is `SIGHUP` (historical convention from `inetd` / `apache` / `nginx`).

### 2.2 Architecture (stdlib-only, MacBook-safe per ADR-023)

```
   process
     ├─ file: /etc/pheno/config.toml (or .yaml, .json)
     ├─ ConfigReloader::new(initial)        ──┐
     │                                          │ shared via Arc
     ├─ request-loop threads:                   │
     │   reloader.current() → Arc::clone ──────┤
     │                                          │
     └─ background "hot-reload" thread: ───────┘
        loop every 250 ms:
          if file mtime changed:
            read_and_install(reloader, path, parse)
              ├─ read_to_string (stdlib)
              ├─ parse (caller closure)
              ├─ on Ok: reloader.install(parsed)
              │         (= *guard = Arc::new(parsed))
              └─ on Err: keep current (last-known-good)
```

The **polling-file-mtime** model is the substrate-level stdlib-only primitive. The daemon-side crate (`pheno-daemon-base` or similar) layers the actual `SIGHUP`-via-`signal-hook::flag` registration on top — keeping `pheno-config` itself stdlib-only and `#![forbid(unsafe_code)]`-clean per ADR-023.

The two-step indirection (flag → background thread, not flag → handler) is deliberate. **Async-signal-safe code is constrained** — you cannot allocate, take a `Mutex`, do file I/O, or call `tracing` from a signal handler. The `flag` approach (`signal_hook::flag`) flips an `AtomicBool` from the handler (the one async-signal-safe operation) and a dedicated thread does the work in normal context. This is the fleet-standard pattern.

### 2.3 Unix-only

The `watch_sighup` function is gated `#[cfg(unix)]`. Windows has no `SIGHUP`; the non-Unix stub returns `ConfigError::UnsupportedPlatform` so cross-compile cases fail loudly at startup instead of silently doing nothing.

### 2.4 Failure paths

| Failure | Behavior |
|---|---|
| File read I/O error | Logged at WARN; previous config retained. |
| Parse error | Logged at ERROR with file path; previous config retained. |
| Signal handler panic | `signal-hook` survives; background thread exits, daemon keeps serving. |
| Background thread panic | `Mutex` poisons; subsequent `current()` calls return `Err`, daemon logs and exits cleanly. |
| Non-Unix platform | `watch_sighup` returns `UnsupportedPlatform` error at startup. |

The poison-detection path is why `hot_reload` uses `Mutex<Arc<T>>` and not raw `Arc::swap` — see §3.

## 3. Atomic-swap proof

### 3.1 The invariant

**Every reader of `ConfigReloader::current()` must see a complete `Arc<T>`, never a torn read.** A torn read at the swap boundary would violate the type system and produce undefined behavior (e.g., reader sees the new `Arc<String>`'s heap allocation but the old `String`'s pointer). The atomic-swap guarantee is the load-bearing invariant.

### 3.2 Two stdlib-only patterns

`pheno-config` uses **two different patterns**, one per module, both stdlib-only, both satisfying the invariant:

| Module | Primitive | Rationale |
|---|---|---|
| `hot_reload` | `Mutex<Arc<T>>` | **Poison detection** — a panicking writer trips the mutex, readers get `Err` instead of silently serving stale config. Per ADR-078 (L52, encryption at rest), silent failure on a corrupt config reload is the failure mode we want to surface loudly. |
| `secret_rotation` | `Arc::swap` (stdlib) | The history-stack invariant needs true atomic pointer swap (no lock, no poisoning — `ZeroizeOnDrop` handles the failure path). |

Both patterns guarantee: a reader either sees the old `Arc<T>` or the new one, never a half-state. Lock contention is negligible: the lock is held for a single `Arc::clone` (nanoseconds), so the hot-path read scales linearly with cores up to ~1 G read/s.

### 3.3 Why not `arc-swap` / `parking_lot` / `crossbeam`?

Per the v22-T4 brief and ADR-023 (MacBook-safe), `pheno-config` pulls **no additional crates** for this track. The substrate stays stdlib-only; `signal-hook` is in the consumer-side daemon crate, not here. This is the substrate placement rule from ADR-023 — pure reusable lib, single concern, single crate.

### 3.4 Integration test proof

`tests/hot_reload_integration_test.rs` is the load-bearing artifact. The `reload_under_load_is_consistent` test:

1. Spawns **8 reader threads**, each loops on `reloader.current()` for 300 ms.
2. From the main thread, calls `install("ConfigB")` at t≈20ms and `install("ConfigC")` at t≈120ms.
3. Joins all 8 readers.
4. Asserts **every reader** observes the sequence `[ConfigA → ConfigB → ConfigC]` with the property that *no reader sees ConfigC without first seeing ConfigB* (which would indicate a torn read where the ConfigA pointer leaked into the ConfigC window).

The test is bounded to ~300 ms so it runs in the CI test budget (< 1 s). A reader that catches a torn read would fail the `saw_c ⇒ saw_b` invariant — the direct failure mode for the atomic-swap guarantee.

A second integration test (`current_returns_arc_with_correct_strong_count`) asserts the `Arc::strong_count` invariant: three clones of the current `Arc` produce strong_count = 4 (1 inside the reloader + 3 live clones). This proves no double-allocation or accidental clone of the inner `T`.

## 4. Secret-rotation flow

### 4.1 Wire protocol

```
   process
     ├─ SecretRotator::new(initial_api_key, FileSource::new("/run/secrets/api-key"))
     ├─ daemon request loop calls rotator.current() — single Arc::clone
     ├─ orchestrator (Vault sidecar / k8s) writes new key to the file
     ├─ daemon calls rotator.rotate()
     │     ├─ source.fetch() -> new ApiKey (or RotationError)
     │     ├─ on Ok: Arc::swap the new key in, push old to history
     │     └─ on Err: keep current, return error
     ├─ downstream service accepts new key (or returns 401)
     │     ├─ accept: trim_history(0) → drop the old (ZeroizeOnDrop fires)
     │     └─ 401:  rotator.rollback() → restore from history, retry
     └─ continue
```

### 4.2 The `RotationSource` trait

```rust
pub trait RotationSource: Send + Sync {
    fn fetch(&self) -> Result<ApiKey, RotationError>;
}
```

Three concrete impls ship in `pheno-config`:

| Source | Backend | Status |
|---|---|---|
| `FileSource` | Local file (k8s `tmpfs` mount) | **Implemented** (stdlib `fs::read_to_string`). |
| `EnvSource` | Process env var (vault-agent) | **Implemented** (stdlib `env::var`). |
| `VaultSource` | HashiCorp Vault (HTTPS + mTLS) | **Stub** — returns "feature not enabled" error. Real impl lives in planned `pheno-vault` consumer crate (v23) so pheno-config stays stdlib-only. |

### 4.3 Rollback

`SecretRotator` holds a `Vec<Arc<ApiKey>>` history stack. On `rotate()`, the *current* secret is pushed to the back; on `rollback()`, the back is popped and swapped in via `Arc::swap`. `trim_history(max_len)` caps the rollback window to a consumer-defined N (typically 1 — most-recent rotation only).

The rollback path is the load-bearing piece for **zero-downtime secret rotation**: if the downstream service rejects the new credential (HTTP 401, DB auth fail), the daemon calls `rollback()` and retries with the previous key — no process restart, no config-file edit, no SIGHUP.

### 4.4 Secret-material hygiene (ADR-078 / L52)

- `ApiKey` newtype is `ZeroizeOnDrop`. When the `Arc<ApiKey>` inside the rotator is dropped (via `trim_history` or a rollback that retires the key), the inner bytes are wiped before the allocator frees the heap.
- `Arc::swap` is used for the pointer exchange (no `unsafe`, no `Arc::make_mut` — the latter would clone the inner bytes and defeat zeroization).
- No secret material ever appears in `Debug`, `Display`, or error messages (the error variants carry *which* source failed, not the key bytes).

## 6. Deviations from the brief

### 6.1 `Cargo.toml` not modified

The brief said "Stdlib-only", which we satisfied by **NOT** adding `signal-hook`, `arc-swap`, `crossbeam-channel`, or `anyhow` as dependencies. The SIGHUP pump in `hot_reload.rs` documents in its module-level rustdoc that the daemon-side wiring (the actual signal handler installation) is a **separate crate** (e.g., `pheno-daemon-base`) that depends on `signal-hook`; the pheno-config substrate exposes the *reload primitive* (`ConfigReloader`, `read_and_install`, file-mtime polling watcher) and lets the daemon crate wire the SIGHUP pump. This keeps `pheno-config` a pure reusable library (ADR-023 substrate placement rule).

### 6.2 `VaultSource` is a stub

The `VaultSource` returns a "feature not enabled" error by design (stdlib-only constraint). The real impl ships in a downstream consumer crate (`pheno-vault`, planned v23) that depends on `vaultrs` or `reqwest` + mTLS. The trait seam (`RotationSource`) is uniform across the three sources so the swap-in is a one-line change at the consumer site.

### 6.3 Polling vs SIGHUP-driven watcher

The substrate-level `watch_sighup` is a **polling file-mtime check** (250 ms tick), not a SIGHUP-via-`signal-hook::flag` driver. The latter belongs in the consumer-side daemon crate per §6.1. The polling model is stdlib-only and works cross-platform; the SIGHUP model is Unix-only and adds a `signal-hook` dependency. Both are documented in the module rustdoc.
