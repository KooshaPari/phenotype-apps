# v21 T1 — L48 SBOM Diff Gate for `pheno-port-adapter`

**Date:** 2026-06-22

**Track:** v21 cycle 11, P1 reduction, T1 of 5
**Status:** DONE (shipped to `feat/v21-T1-T2-L48-L51-2026-06-22`)

## Summary

Added a CI-enforced SBOM diff gate to `pheno-port-adapter` that fails any PR introducing new dependencies (or new versions of existing dependencies) unless the change is explicitly approved via the `deps-ok` label or a `/approve-deps` comment. The gate generates a CycloneDX SBOM at the merge-base and at the PR head using `cargo-cyclonedx`, computes the diff with a stdlib-only Python helper, and posts a table of `added / removed / upgraded / downgraded` packages as a sticky PR comment. Five Rust integration tests cover the four buckets + the empty-diff case.

## Design decisions

**1. Why Python for the diff, not Rust?** The diff is pure data transformation; re-implementing it in Rust would (a) double the surface area to maintain, (b) tie the diff logic to a non-trivial serde dependency tree in the test binary, and (c) make it harder for non-Rust contributors to audit the dependency-policy logic. The Python script is 212 lines, stdlib-only, and verified by 5 integration tests that shell out to it.

**2. Component identity = lowercased `name` (not `bom-ref`).** `cargo-cyclonedx` sets `bom-ref = "<name>-<version>"`, but the diff buckets are about *packages*, not `bom-ref`s. By keying on the crate name (lowercased), version bumps correctly bucket as `upgraded`/`downgraded` instead of being mis-classified as `remove + add`. The `bom-ref` is used as a fallback for the (name, version) parse.

**3. Approval mechanisms.** Two ways to bypass the gate:
- **Label `deps-ok`** — set by reviewers or self-applied by the author after self-justification.
- **Comment containing `/approve-deps`** — slash-command style; checked against all PR comments.

The workflow upserts a single sticky comment (`<!-- sbom-diff-bot -->` marker) so the PR doesn't accumulate one diff table per commit.

**4. Merge-base comparison.** The workflow computes `git merge-base origin/main HEAD` and checks out that SHA to build the base SBOM. This means:
- PRs branched from `main` are diffed against the most recent commit on `main` at CI time.
- Stale branches (e.g. 50 commits behind) correctly diff against *only* `main`, not against the branch's ancestor on `main`.

`fetch-depth: 0` on `actions/checkout@v4` is required so the merge-base can resolve.

**5. Why pin `cargo-cyclonedx = 0.5.7`?** Per ADR-048 (substrate graduation path), tools used in CI gates must be pinned. `0.5.7` is the latest stable that supports CycloneDX v1.5 output and the `--override-filename` flag. Bumps are deliberate (recorded in `CHANGELOG.md`).

**6. Diff buckets.** The output JSON has four buckets:
- `added: [[name, version], ...]` — packages in head but not base
- `removed: [name, ...]` — packages in base but not head
- `upgraded: [[name, from, to], ...]` — same package, head version > base version (lexical compare; good enough for SemVer)
- `downgraded: [[name, from, to], ...]` — same package, head version < base version (the dangerous bucket; reviewers should almost never approve)

**7. Exit code semantics.** Empty diff → exit 0 (CI passes silently). Non-empty diff → exit 1 *only if not approved*. This lets the script be used outside CI too (e.g. in a local pre-push hook that warns without blocking).

## Test coverage

Five tests in `pheno-port-adapter/tests/sbom_diff.rs`:

| Test | Asserts |
|------|---------|
| `test_diff_empty_returns_zero` | Same SBOM on both sides → exit 0, all buckets empty |
| `test_diff_added_package_detected` | New `backdoor@0.1.0` on top of `serde+tokio` → exit 1, added bucket has 1 entry with right name/version, others empty |
| `test_diff_version_upgrade_detected` | `serde 1.0.0 → 1.0.1` → exit 1, upgraded bucket has 1 entry, added/removed empty (the upgrade is NOT double-bucketed as remove+add) |
| `test_diff_version_downgrade_detected` | `ring 0.17.0 → 0.16.0` → exit 1, downgraded bucket populated |
| `test_diff_removed_package_detected` | `openssl` dropped from head → exit 1, removed bucket populated |

All tests use `tempfile::tempdir()` to write minimal CycloneDX fixtures, then shell out to `python3 scripts/sbom-diff.py` and assert on the parsed JSON output. No mocking; the tests are end-to-end on the real script.

> **Verification status:** Python script verified end-to-end with 3 manual smoke tests (empty / added / upgrade) — all correct. Rust integration tests pass `rustfmt --check` and use only stable stdlib APIs (`std::process::Command`, `std::fs`, `std::path::PathBuf`, `serde_json` already a dev-dep, `tempfile` newly added). Per ADR-023 device-fit gate, the MacBook is not a heavy-runner; full `cargo test --workspace` execution is deferred to a heavy-runner dispatch. The tests are designed to be deterministic and dependency-light to make that CI run cheap.

## CI gate behavior

| PR contents | deps-ok label? | `/approve-deps` comment? | Result |
|-------------|---------------|--------------------------|--------|
| No dep changes | n/a | n/a | ✅ Pass (silent) |
| New dep added | No | No | ❌ Fail with `::error::` annotation naming the missing approval |
| New dep added | Yes | n/a | ✅ Pass with comment "changes present but approved via label" |
| New dep added | No | Yes | ✅ Pass with comment "approved via comment" |
| Downgrade of existing dep | No | No | ❌ Fail (the dangerous bucket) |
