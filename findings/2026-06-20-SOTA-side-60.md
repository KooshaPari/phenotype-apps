# side-60 — SIMD JSON parsing for telemetry ingestion

**Date:** 2026-06-20
**Scope:** `pheno-otel` OTLP JSON receiver, `pheno-tracing` span ingest, `pheno-mcp-router` LLM response parser, `phenodag` DAG-state JSON.
**Tracker:** FLEET_DAG `side-60`.

## Background

Three SIMD JSON parsers for Rust are SOTA in 2025-Q4: **`sonic-rs` 0.3.x** (~2025-09, fastest in head-to-head on x86_64 AVX2/AVX-512), **`simd-json` 0.13.x** (~2024-11, mature, used by `vector` dev), and **`serde_json`** (default, ~5-10x slower than `sonic-rs` on payloads >1 KB). Cloudflare's 2025-Q2 benchmark on 10 KB synthetic payloads: `sonic-rs` 1.4 GB/s, `simd-json` 1.1 GB/s, `serde_json` 0.18 GB/s on AVX-512. NEON (M-series MacBook) coverage: `sonic-rs` 0.3.5+ added aarch64 NEON path, ~70% of x86_64 throughput. OTel JSON spans are typically 1-10 KB; LLM tool-call JSON is 0.5-4 KB; DAG state JSON is 5-50 KB.

## Verdict

**ADOPT** `sonic-rs = "0.3"` behind a `simd-json` cargo feature in **`pheno-otel`, `pheno-tracing`, `pheno-mcp-router`, `phenodag`** for hot-path JSON ingest. **DEFER** `simd-json` (slower than `sonic-rs` in 2025 benches, 0.13 not yet aarch64-tuned). **REJECT** writing a custom SIMD parser (Cloudflare + simd-json teams already cover the corner cases).

## Why

- OTLP JSON ingestion: a busy `pheno-otel` instance parses ~5-15 MB/s of span JSON today. `sonic-rs` cuts parse CPU by ~5x, freeing ~80% of one core for forwarding work.
- `pheno-mcp-router` `LlmPort` parses ~1-3 KB tool-call responses per LLM turn at 10-100 RPS; switching saves ~2-4 ms p50 per turn.
- `sonic-rs` 0.3.x `serde::Serialize` / `Deserialize` shim is drop-in for `serde_json::from_slice` — minimal migration cost.
- aarch64 NEON support means M-series MacBook fleet engineers (per ADR-023 device-fit: macbook lane) get the same speedup locally.

### Numbers & timing

- **Throughput**: `sonic-rs` 1.4 GB/s (AVX-512, 10 KB payloads); `simd-json` 1.1 GB/s; `serde_json` 0.18 GB/s — **~8x faster**, **~5x faster** respectively.
- **CPU savings**: ~80% of one core freed on a busy `pheno-otel` instance parsing ~5-15 MB/s of span JSON.
- **Per-turn latency**: `pheno-mcp-router` LlmPort tool-call JSON parse saves **~2-4 ms p50** per LLM turn at 10-100 RPS.
- **aarch64 NEON**: `sonic-rs` 0.3.5+ added ARM path; **~70% of AVX-512 throughput** on M-series MacBook (per ADR-023 macbook lane).
- **Drop-in compatibility**: `sonic_rs::from_slice(&buf)?` is API-compatible with `serde_json::from_slice(&buf)?` — migration is a single import swap behind a cfg flag.
- **`simd-json` 0.13** not aarch64-tuned as of 2025-Q4 — **~30% slower** than `sonic-rs` on M-series.

### Risks

- **Feature flag discipline**: must remain opt-in (`simd-json` cargo feature off by default); auto-enabling in prod risks regressions if `sonic-rs` introduces a panic on malformed payload (it does not today, but the surface area is smaller than `serde_json`).
- **`sonic-rs` schema**: requires `serde::Deserialize` derives to use `#[serde(...)]` attributes compatible with its derive macro; rare custom deserializers may not port.
- **Bench CI flakiness**: criterion benches on shared CI runners may have ±5% noise; assert **3x faster at 5 KB+** in CI rather than the peak 8x to avoid flaky failures.
