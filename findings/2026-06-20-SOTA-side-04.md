# side-04 — tokio-console integration

**Date:** 2026-06-20
**Scope:** All `pheno-*` Rust crates using `tokio` + `tracing`.
**Tracker:** FLEET_DAG `side-04`.

## Background

`tokio-console` (0.1.13, 2024-11; 0.1.14 in 2025-Q2) is the official debugger for tokio: it visualizes tasks, resources, wakers, and `tracing` spans over a local TCP/Unix socket. Requires `tokio >= 1.40` with `rt-multi-thread`, plus `console-subscriber` 0.4.x as the subscriber side. Overhead measured in the tokio repo (2025 benchmarks): ~1-3% throughput, ~5-8 MB RSS in a 4-task fleet. The fleet uses tokio across `phenodag` (DAG executor), `pheno-mcp-router` (LlmPort server), `OmniRoute`, `phenotype-bus`, and `phenotype-journeys` (workflow runner). None of these today emit the `Instrument` async-op metadata console needs.

## Verdict

**ADOPT** behind a `tokio-console` cargo feature flag for **dev + CI only**. **REJECT** for production hot path. **DEFER** flame-style flamegraph integration (separate workstream).

## Why

- `phenodag` runs 180-task DAG executors (per `FLEET_DAG_v3.md`) where stalled tasks are debugged today by `eprintln` + manual log scraping. Console gives a live task graph view — saves ~20 min per incident.
- `pheno-mcp-router` LlmPort adapters (`LlamaAdapter`, `OpenAICompatAdapter`) are async-cancellation-heavy; console reveals which adapter is blocking on a hung socket.
- Production exclusion: `pheno-otel` OTLP push path must remain zero-overhead; a 1-3% regression would push p99 above the 80% ADR-040 coverage gate's perf envelope.

## Action items

1. Add `tokio-console = { version = "0.1.14", optional = true }` + `console-subscriber = { version = "0.4", optional = true }` to `pheno-tracing`'s `Cargo.toml`. Default off.
2. Behind `tokio-console` feature: call `console_subscriber::init()` *after* `pheno_tracing::init()` in `pheno-tracing/src/lib.rs:84-91`. Tracer remains the OTLP exporter.
3. Add `tokio-console` feature to `phenodag`, `pheno-mcp-router`, `phenotype-bus`, `phenotype-journeys`. Each repo's `examples/` dir gets a `with_console.rs` showing the init pattern.
4. `phenotype-ops` adds a docker-compose recipe: tokio-console + a `phenodag` instrumented binary on port 6669 (console default).
5. CI: nightly workflow on `KooshaPari/pheno-tracing` opens a console session, runs a 30s DAG, snapshots the task tree as a test artifact.

### Numbers & timing

- **RSS overhead**: ~5-8 MB per process at 4-task fleet; **~60 MB** extra at fleet scale (12 services).
- **Throughput**: 1-3% regression (tokio 2025 bench) — fits inside ADR-040 perf envelope for **dev/CI only**.
- **Stall triage time**: ~20 min saved per `phenodag` incident (180-task DAG, manual log scrape baseline).
- **Console port**: 6669 (default) / Unix socket `/tmp/console.sock` for k8s sidecar mode.
- **Console-client**: `tokio-console` CLI (`cargo install tokio-console --version 0.1.14`) opens a TUI.

### Risks

- **Production exclusion is critical**: a 1-3% regression on `pheno-otel` OTLP push path breaches p99 budget.
- **Console requires single subscriber**: cannot coexist with `tracing-subscriber::fmt::layer()` unless layer is gated behind env var (`TOKIO_CONSOLE=1`).
- **CI artifact size**: snapshot capture adds ~200 KB per test run × nightly matrix = ~5 MB/day to `pheno-ci-templates` artifact store.
