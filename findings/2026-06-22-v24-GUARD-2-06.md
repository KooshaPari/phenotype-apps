# Guardrail: SLO burn-rate alert windowing policy (GUARD-2-06)

**Date:** 2026-06-22
**Owner:** observability circle
**Pillar:** L63 (Observability — SLO burn-rate alerts)
**Status:** Proposed

## Current state

The v24 T5 plan authors 4 multi-window burn-rate alerts (1h fast-burn, 6h
fast-burn, 6h/24h slow-burn, 3d slow-burn) per Google SRE ch. 5. Without an
enforced policy on burn-rate thresholds, individual services pick ad-hoc
values (one team uses 2x, another 14x) and the on-call gets either paged for
spurious noise or never paged at all. ADR-091 codifies the windows but not
the multipliers, so drift creeps in via copy-paste of older rules.

## Proposed guardrail

A `slo-burn-rate.schema.json` in `phenotype-ops/grafana/schemas/` that
validates every alert YAML in `phenotype-ops/grafana/alerts/*.yaml` against
required fields: `window`, `burn_rate_threshold` (must be in {2, 6, 14, 28}
set per SRE ch. 5), `for` (must be ≤ window/4), `severity` (page | ticket).
PR validation via `jsonschema` step in CI. Reject any alert that doesn't
match; require an ADR override comment to use a non-standard multiplier.

## Concrete implementation

`phenotype-ops/grafana/schemas/slo-burn-rate.schema.json` (~60 LOC):

```json
{
  "type": "object",
  "required": ["alert", "expr", "for", "labels", "annotations"],
  "properties": {
    "labels": {
      "properties": {
        "burn_rate_threshold": { "enum": [2, 6, 14, 28] },
        "severity": { "enum": ["page", "ticket"] }
      }
    },
    "for": { "pattern": "^[0-9]+[smh]$" }
  }
}
```

`.github/workflows/slo-lint.yml` (~25 LOC):

```yaml
- run: pip install jsonschema
- run: python scripts/lint-slo-alerts.py alerts/*.yaml
```

## Effort

**S** — schema + lint script + CI step. ~half-day on MacBook. Backward
compatible: existing alerts get linted but blocking is enabled after one
weekly-cadence grace period (per ADR-041).
