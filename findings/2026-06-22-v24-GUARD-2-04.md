# Guardrail: Connection pool exhaustion detector (GUARD-2-04)

**Date:** 2026-06-22
**Owner:** performance circle
**Pillar:** L18 (Performance — connection pool tuning)
**Status:** Proposed

## Current state

Substrate crates that talk to Postgres / Redis / Vault use `deadpool` /
`bb8` pools with default sizes (10 conns). When a slow downstream causes
request queueing, the pool fills, new requests block on `.acquire()` for
minutes, and the service appears to hang. There is no signal until the
client times out (default 30s) and the user reports the issue. By the time
the alert fires, hundreds of requests are stuck.

## Proposed guardrail

Every pool-backed substrate crate wraps `.acquire()` in a `pheno-pool-metrics`
instrumented future that records `acquire_wait_seconds` (histogram) and
`pool_in_use` (gauge). A `PoolExhaustion` alert fires when
`histogram_quantile(0.99, acquire_wait_seconds) > 5s` for 2 minutes (warning
threshold) or `> 10s` for 1 minute (page threshold). The pool also exposes a
`Pool::try_acquire_with_budget(timeout)` that fails fast instead of hanging.

## Concrete implementation

`pheno-pool-metrics` crate (~120 LOC) — generic over `deadpool::Pool<T>` and
`bb8::Pool<T>`:

```rust
pub struct InstrumentedPool<P> { inner: P, metrics: PoolMetrics }
impl<P> InstrumentedPool<P> {
    pub async fn acquire(&self) -> Result<Conn, AcquireError> {
        let start = Instant::now();
        let conn = timeout(POOL_ACQUIRE_BUDGET, self.inner.acquire()).await??;
        metrics.observe_acquire(start.elapsed());
        Ok(conn)
    }
}
```

Alert rule (`alerts/pool-exhaustion.yaml`):

```yaml
- alert: PoolExhaustion
  expr: histogram_quantile(0.99, sum by (service, le) (rate(acquire_wait_seconds_bucket[5m]))) > 5
  for: 2m
```
