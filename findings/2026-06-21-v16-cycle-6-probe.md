## Blockers

1. **T3 (latency budgets in CI)** — MacBook can author the `perf-budgets.toml` extension (~50 LoC) but the validation step requires running `cargo bench -- --bench load` against the cycle-1 cohort (~15 min wall per crate on heavy-runner). Heavy work per ADR-023 Rule 2.1 → `device: heavy-runner`. **Unblock:** dispatch to a subagent with the heavy-runner profile; can run all 3 crates in parallel.
2. **T4 (cost optimization)** — `pheno-cost-card` is currently v0.1.x and the cost formula `cost = histogram_quantile(p99) × cloud_pricing[region]` is **not yet implemented**; needs `pheno-cost-card` v0.2.0 first. **Unblock:** confirm v0.2.0 release on `KooshaPari/pheno-cost-card` before T4 starts (check `crates.io` for the release date).
3. **T7 (chaos CI gate)** — chaos scenarios (random process kill, latency injection via `tc/netem`, partition via `iptables`) require Linux kernel features unavailable in Codespaces devcontainers. **Unblock:** restrict chaos workflow to self-hosted Linux runners (`runs-on: [self-hosted, chaos]`). Verify the runner pool has the `chaos` label before T7 starts.
4. **T8 (release artifacts)** — `cosign` keyless signing requires GitHub OIDC trust configured for `KooshaPari` repos. **Unblock:** verify with `gh api /repos/KooshaPari/phenotype-router/actions/oidc-customization/sub` before T8 starts. Fallback: KMS key.
5. **T9 (e2e tests + cycle-break refactor)** — `pheno-port-adapter` has 3 circular dep cycles (v15 T1 audit); breaking them is a prerequisite for the e2e harness to compile cleanly. **Unblock:** trait extraction is a separate refactor; either fold into T9 (risky) or do as a v16+1 prerequisite.

## Risks & Caveats

1. **T3 latency budgets may be too tight.** Single-run p99 measurements are noisy; using a single measurement as the gate will cause CI flakes. **Mitigation:** use 95th percentile of last 10 runs as baseline (same pattern as v15 T2 criterion).
2. **T4 cost attribution requires fresh cloud pricing data.** Pricing tables go stale quarterly; if `pheno-cost-card` is pinned to an old release, attribution drifts. **Mitigation:** pin pricing table to release; bump quarterly; alert on >30-day-old pricing.
3. **T7 chaos tests can damage runner state.** Random process kill + iptables partition can leave runners in degraded state if the cleanup step fails. **Mitigation:** run in isolated namespace (`--network=none`); restore snapshot on completion; post-mortem template auto-generated on failure.
4. **T8 cosign keyless signing requires GitHub OIDC.** Without OIDC trust, cosign falls back to KMS key (less auditable but functional). **Mitigation:** verify OIDC on first run; fall back to KMS if unavailable; document fallback in `release.yml`.
5. **T9 cycle-break refactor is risky.** Breaking the 3 circular dep cycles in `pheno-port-adapter` is a non-trivial refactor (touching 3 module pairs: `adapter/tcp`, `adapter/http`, `adapter/test`). **Mitigation:** do the refactor as a v16+1 prerequisite (separate small PR), then T9 builds the e2e harness on the refactored codebase.
6. **T10 regression threshold of 5% may be too tight.** Cycle-1 cohort's `pheno-port-adapter` is sensitive to compiler-version perf changes; 5% may block valid improvements. **Mitigation:** allow one-time override with ADR; default 10% in v16, tighten to 5% in v17 once we have 4 weeks of baseline data.
7. **L57 substrate (`pheno-otel::metrics`) requires an ADR-040 amendment.** The current ADR-040 substrate-quality-bar doesn't include a metrics gate. The amendment is on the cycle-5 backlog (due 2026-06-22). If it slips, v17's L57 closure slips with it. **Mitigation:** track ADR-040 amendment as a hard v17 dependency; surface in Monday 2026-06-22 weekly review.
