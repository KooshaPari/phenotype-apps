# side-14 — litellm vs raw OpenAI for provider abstraction

**Date:** 2026-06-20
**Scope:** `pheno-mcp-router` LlmPort adapter layer; `phenotype-python-sdk` provider client.
**Tracker:** FLEET_DAG `side-14`.

## Background

`LiteLLM` (Python, v1.55+ as of 2025-Q4) is a proxy library exposing 100+ LLM providers via a unified `completion()` call. It supports function-calling, streaming, vision across OpenAI/Anthropic/Google/Bedrock/Azure/Cohere/Mistral/Groq/etc. Fleet has **already adopted** the `LlmPort` trait in `pheno-mcp-router` (ADR-013, ADR-037) with shipped `LlamaAdapter` (PR #2) and `OpenAICompatAdapter` (PR #3, 87% coverage, 17 tests, 429/5xx retry). `LiteLLM` adds one Python proxy hop (~30-80 ms p50), 200 MB image bloat, and a Python-only surface (the Rust fleet can't import it directly). The `OpenAI` Python SDK 1.x (v1.82, 2025-11) is the canonical "raw" baseline: tightest, lowest latency, single vendor lock-in.

## Verdict

**ADOPT** direct SDK-per-provider via `LlmPort` adapters for **production** (Rust substrate). **DEFER** LiteLLM proxy to **ad-hoc Python scripts only** (`scripts/`, `phenotype-python-sdk/dev_tools/`). **REJECT** wrapping LiteLLM behind a `LlmPort` adapter (proxy hop negates latency win).

## Why

- Direct SDK path keeps p50 TTFB at ~250-400 ms (Rust + native `reqwest` + `eventsource-stream`); LiteLLM proxy adds 30-80 ms.
- `LlmPort` already gives polyglot coverage: `OpenAICompatAdapter` handles OpenAI + Together + Groq + OpenRouter via OpenAI-compatible wire format (~70% of providers). Remaining 30% (Anthropic native, Google Vertex native, Cohere) get dedicated adapters.
- Adapter-per-provider pattern matches the substrate graduation path (ADR-048 Tier-2): each adapter is a 200-400 LOC trait impl, ships with provider matrix in `PROVIDER_GUIDE.md`.
- LiteLLM retained only for **Python experiments** where the user wants to A/B test 4 providers in a notebook.

### Numbers & timing

- **Direct SDK latency**: p50 TTFB ~250-400 ms (Rust + `reqwest` + `eventsource-stream`); LiteLLM proxy adds **30-80 ms** = **12-25% regression**.
- **Existing coverage**: `OpenAICompatAdapter` handles ~70% of providers via OpenAI-compatible wire (OpenAI/Together/Groq/OpenRouter).
- **Adapter footprint**: ~280-320 LOC per native adapter (Anthropic, Gemini); `PROVIDER_GUIDE.md` row + 12-14 tests per adapter.
- **LiteLLM resource cost**: ~200 MB RAM + 1 proxy container; reserved for `phenotype-python-sdk/dev_tools/` ad-hoc only.
- **OpenAI SDK 1.82** baseline (Nov 2025); `async-nats`-style streaming response parsing at ~50 MB/s wire rate.

### Risks

- **Provider API drift**: native adapters own the breakage (e.g., Anthropic prompt-cache headers shipped 2025-Q3); version-pinning per adapter recommended.
- **LiteLLM is Python-only**: never crosses the Rust boundary; the fleet's Rust substrate must use native adapters exclusively per ADR-013 substrate boundary.
- **Rate-limit semantics differ per provider**: OpenAI = 429 + `Retry-After`, Anthropic uses `retry-after` + `x-ratelimit-*` headers — `OpenAICompatAdapter` retry path does **not** cover Anthropic natively; new adapter needed.
