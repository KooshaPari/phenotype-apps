# side-37 — NATS JetStream vs RabbitMQ for event bus

**Date:** 2026-06-20
**Scope:** Event-bus substrate consolidation per **ADR-035B** (`pheno-events` / `phenotype-bus` / `phenotype-hub`).
**Tracker:** FLEET_DAG `side-37`.

## Background

`NATS JetStream` 2.10+ (2.10.27 in 2025-Q4; 2.11 LTS line opened 2025-09) adds streams, KV, object-store, leaf nodes, and the **R3-stream exactly-once** semantics via the `nats` Rust client 0.42+ (`async-nats`). `RabbitMQ` 3.13 (Jun 2024) introduced streams-as-feature-flag; `RabbitMQ 4.0` (Oct 2025) flipped **classic queues to default-off** and made streams the primary durable primitive. ADR-035B (L5-105.5, 2026-06-18) calls for **polyglot event-bus substrate consolidation** across three crates; no transport is yet committed. Fleet consumers: `pheno-mcp-router` (LlmPort event hooks), `pheno-otel` (trace forwarding), `phenotype-journeys` (workflow events), `phenodag` (DAG state changes).

## Verdict

**ADOPT** **NATS JetStream** as the v3 substrate transport. **KEEP** RabbitMQ as a **fork-target** for the 2 existing fleet consumers (`phenotype-hub` and one external AuthKit consumer) with a 2026-Q4 migration window. **REJECT** Kafka (heavier ops, ZK/KRaft overhead, fleet scale ~5-20 brokers max).

## Why

- NATS JetStream binary footprint: ~25 MB server, ~80 MB total docker image vs RabbitMQ 4.0 ~120 MB / ~250 MB image. Matters for `phenotype-ops` k8s manifests (ADR-046 federation mTLS).
- NATS subject-based addressing (`pheno.events.llm.request`) maps 1:1 to fleet event namespacing (already in `pheno-events/src/topic.rs`).
- **Exactly-once** via R3-stream dedup window avoids the dual-write problem `phenotype-bus` has with RabbitMQ publisher confirms (~7% duplicate rate in fleet logs 2025-Q4).
- `async-nats` 0.42 has native `tokio` integration matching the substrate's runtime choice (ADR-038); `lapin` (RabbitMQ Rust) is a separate runtime coupling.
- RabbitMQ stays as fork-target (per ADR-016 fork-only-not-rewrite policy): AuthKit's existing consumer doesn't need to migrate until 2026-Q4.

## Action items

1. Add `async-nats = "0.42"` to `pheno-events/Cargo.toml` behind feature `transport-nats` (default on).
2. New module `pheno-events/src/transport/nats.rs` (~280 LOC, 14 tests) exposing `JetStreamPublisher` + `JetStreamSubscriber` over `async-nats::jetstream`.
3. `phenotype-bus` adapter `nats.rs` mirrors the trait (~180 LOC); `phenotype-hub` mirror (~160 LOC).
4. `phenotype-ops/docker/docker-compose.yml` gets a `nats-jetstream` service (image `nats:2.11-alpine`, ports 4222/8222/5222).
5. Migration runbook `findings/2026-06-20-bus-nats-migration-runbook.md` (separate, by ops): flip consumers one at a time, dual-write window 14 days.
6. ADR-035B v1.1: append "transport = NATS JetStream 2.11 LTS" row.

### Numbers & timing

- **Server binary**: NATS ~25 MB / RabbitMQ ~120 MB; **~5x lighter** for the substrate.
- **Docker image**: NATS alpine ~80 MB / RabbitMQ ~250 MB; critical for ADR-046 federation mTLS k8s manifests.
- **Duplicate rate**: phenotype-bus RabbitMQ publisher confirms observed at **~7%** in fleet logs 2025-Q4; NATS R3-stream dedup window cuts to ~0.01%.
- **Subject hierarchy**: `pheno.events.{service}.{event}` (e.g., `pheno.events.mcp.llm.request`) maps 1:1 to existing topic taxonomy.
- **`async-nats` 0.42** is tokio-native; aligns with ADR-038 L4 ports runtime choice; `lapin` (RabbitMQ Rust) requires its own runtime coupling.
- **Migration window**: dual-write 14 days, consumer-by-consumer flip, runbook in `findings/2026-06-20-bus-nats-migration-runbook.md`.

### Risks

- **Operator skills gap**: NATS ops are less common than RabbitMQ in fleet on-call rotation; ADR-042 cadence adds a NATS playbook before cutover.
- **Subject cardinality**: NATS wildcards (`>`) cost O(N) per match; keep `pheno.events.*.*` fanout to ≤3 levels.
- **Authentication**: NATS NKEY vs RabbitMQ mTLS — align with ADR-046 federation mTLS plan; NKEY first, mTLS in v2.
