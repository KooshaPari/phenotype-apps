# L28 — Dependency hygiene & CVEs

**Owner:** forge-A15 (security+DX)
**Bloc scope:** AgilePlus + thegent + Tracely + Tracera (4 repos, ~76 crates)
**Cross-cuts:** L19 (supply-chain SBOM/CVE), L10 (CI SHA-pinning), L11 (quality gates)

## Scope

End-to-end dependency hygiene of the bloc: lockfile presence + freshness, Renovate / Dependabot coverage and grouping, **SBOM emission in CI** (CycloneDX/SPDX), **active CVE scanning** (cargo-audit / osv-scanner / Trivy / CodeQL dep findings), license-compatibility allowlists (cargo-deny `[licenses]`), and **pinned vs ranged** dep declarations in `Cargo.toml`. Distinct from L19 (which is bloc supply-chain primitives via `phenotype-dep-guard/`) — L28 is per-repo operational hygiene.

## SOTA 2026

- **Lockfiles are the source of truth** — `Cargo.lock` / `uv.lock` / `pnpm-lock.yaml` / `package-lock.json` / `go.sum` all committed; `cargo build --locked`, `npm ci`, `go build -mod=readonly` in CI to fail on drift.
- **Lockfile freshness ≤ 30 days** between `Cargo.lock` last commit and HEAD — stale lockfiles silently allow untracked vulnerability windows.
- **Single update bot per repo** — Renovate *or* Dependabot, not both. Renovate: `packageRules` + `rangeStrategy: bump` + group non-major updates. Dependabot: `groups:` + per-ecosystem schedule. Mixing them creates duplicate PRs and review noise.
- **SBOM in CI for every release** — CycloneDX 1.5+ (and SPDX 3.0 for fed consumers), attached to GH release, consumed by `dependency-graph` + Dependabot for vulnerability surfacing. Must be generated in a CI job, not a one-off script.
- **CVE matching ≥ 2 scanners** — RustSec via `cargo-deny advisories`, OSV.dev via `osv-scanner`, GHSA via Dependabot security advisories. Different feeds reconcile to the same advisory ID.
- **`cargo-deny [licenses]`** with `confidence-threshold = 0.8` and a version = 2 schema; allowlist reflects project license (`MIT` / `MIT OR Apache-2.0`) and weak-copyleft (MPL-2.0, LGPL-2.1) where compatible.
- **Pinning style** — `Cargo.toml` may use `version = "1"` (range, allows minor upgrades) but `Cargo.lock` pins exact versions; CI must run `cargo update --locked` rejects. For security-critical crates, `[patch.crates-io]` + `cargo-deny [bans] multiple-versions = "warn"` enforces single resolution.
- **Bloc exemplar:** `phenotype-dep-guard/src/sbom.rs:1-131` + `phenotype-dep-guard/.github/workflows/reusable-dep-guard.yml:1-67` (federated `workflow_call`) — but as of L28 audit, **no bloc repo has `uses:` this workflow**.
