## 1. Executive Summary

phenorust is a Rust-language component in the Phenotype fleet (sibling to `pheno-errors`, `pheno-tracing`, `pheno-cargo-template`, `pheno-zod-schemas` on the data/contract side; sibling to `phenotype-voxel`, `phenotype-bus`, `phenotype-otel` on the runtime side). The crate is **not yet at 1.0** because the public API ships with numerical-correctness hazards: 50 sites in the current tree can produce `NaN`, `±Inf`, or a `panic!` on adversarially valid inputs. The 1.0 release must close all 50 sites, lock the public API, and pass a numerical-fuzzing gate.

This plan defines a four-phase release (alpha → beta → rc → stable), per-phase acceptance criteria, a wall-clock estimate, and the coordination protocol with the `phenorust-wtrees` worktree family.

## 3. Current State (recoverable from context)

Based on the Phenotype fleet's documented conventions and the defect class:

- **Language / edition:** Rust, edition 2021 (matching `pheno-cargo-template`, `pheno-errors`, `pheno-tracing`).
- **Build system:** Cargo workspace member; expected to be a library crate with optional `serde`, `rayon`, and `wasm-bindgen` features.
- **Numerical surface:** Float arithmetic, division, log/exp/sqrt, statistical reductions, vectorised kernels — the defect classes that produce NaN/Inf/panic.
- **Panic surfaces (typical for the class):**
  - `unwrap()` / `expect()` on `Result` from fallible numeric conversions (e.g. `f64::from` over `u64` overflow).
  - Index-based access (`vec[i]`) on user-supplied length.
  - `.unwrap()` on a parser that can fail on edge inputs.
- **NaN/Inf surfaces (typical for the class):**
  - `0.0 / 0.0`, `x - x` cancellation, `(-1.0_f64).sqrt()`, `(-x).ln()`, `exp(huge)`, `log(0.0)`.
  - Mean/var reductions on empty slices (NaN from `0.0/0.0`).
  - Softmax / log-sum-exp with large dynamic range (overflow to `Inf`).
- **MSRV:** assumed `1.75.0` to match fleet-wide pinned toolchain (`rust-toolchain.toml`).
- **Public API:** not yet SemVer-locked; pre-1.0 versions currently iterate freely.

## 4. What's Blocking 1.0

### 4.1 Blocker taxonomy

The 50 sites decompose into four classes. The exact split is to be confirmed from the source map; the table below is the working estimate consistent with the stated total.

| Class | Estimated count | 1.0-blocking? | Phase resolved |
|---|---:|:---:|:---:|
| **`NaN`/`Inf` propagating from public API** | ~25 | Yes | Alpha |
| **`panic!` on adversarial-but-valid input** | ~15 | Yes | Alpha |
| **`unwrap`/`expect` in library code (no `Result` surface)** | ~8 | Yes | Beta |
| **Numerical accuracy below documented tolerance** | ~2 | Conditional | RC |

### 4.2 Blocking defects — definition of done

A site is considered *closed* only when all of the following hold:

- [ ] The site has a unique ID (`PHE-####`) in the inventory ledger.
- [ ] The fix is implemented behind the public API (no new panics, no `NaN`/`Inf` leaks to callers).
- [ ] A regression test exists that reproduces the original failure mode on a minimal input.
- [ ] A property test (`proptest`) covers the input neighbourhood.
- [ ] A `cargo fuzz` target covers the input neighbourhood (where the surface is fuzzable).
- [ ] The CHANGELOG entry is filed under "Unreleased → Fixed (1.0 candidates)".
- [ ] A code owner has signed off (per `CODEOWNERS`).

### 4.3 Non-defect blockers

In addition to the 50 sites, the following must land before stable:

- [ ] Public API is frozen; `cargo public-api` diff is empty vs the RC tag.
- [ ] `cargo doc` produces zero warnings; all public items have doc comments.
- [ ] `#![forbid(unsafe_code)]` (or audited `unsafe` if the crate uses SIMD).
- [ ] `cargo deny` clean; MSRV pinned; `Cargo.lock` committed.
- [ ] Fuzz harness runs for ≥1 hour with no crashes (or 24h for `*-rc.1`).
- [ ] A migration guide exists for any pre-1.0 caller.

## 5. Release Phases

### 5.1 Phase 0 — Setup (one-time, pre-alpha)

**Goal:** establish the artifact, the inventory ledger, and the worktree family so that all four release phases have a place to land their work.

- [ ] **Recover the source map.** Locate or regenerate `phenorust-map-v2.md`. If the file truly does not exist, regenerate it via a fresh sweep: `cargo geiger`, `cargo clippy -- -D warnings`, `cargo fuzz list`, and a manual audit of `unwrap`/`expect`/floating-point ops.
- [ ] **Author the 50-site inventory.** Convert the recovered map into a machine-readable ledger (`inventory/1.0-blockers.json` or `.csv`) with columns: `id`, `file`, `line`, `defect_class`, `severity`, `owner`, `status`, `phase_target`, `test_ref`.
- [ ] **Provision `phenorust-wtrees/`.** Create the worktree-family root following the fleet convention. Initial children: `phenorust-wtrees/alpha/`, `phenorust-wtrees/beta/`, `phenorust-wtrees/rc/`, `phenorust-wtrees/stable/`.
- [ ] **Create `.audit/`.** Establish the audit-artifact directory. This plan is its first file; sibling artifacts (`1.0-blockers.json`, `api-baseline.txt`, `fuzz-report-*.md`) will land here.
- [ ] **Lock the MSRV and toolchain.** Pin `rust-toolchain.toml`; record in the source map.
- [ ] **Add CI gates.** Add a job that runs `cargo test`, `cargo clippy -- -D warnings`, `cargo doc --no-deps`, and a 10-minute `cargo fuzz` smoke run on every PR.

### 5.2 Phase 1 — Alpha (≈3 weeks)

**Tag:** `v1.0.0-alpha.1` (and `.2` as needed)

**Goal:** close every NaN/Inf/panic site that has a clear fix. No API changes are allowed after the alpha tag without an `alpha.N` bump.

**Scope:**

- [ ] Triage the 50 sites; freeze the inventory; assign owners.
- [ ] Close all ~25 NaN/Inf sites with explicit return types (`Result<T, PhenorustError>`) or documented sentinel values.
- [ ] Close all ~15 panic-on-valid-input sites by replacing `unwrap`/`expect` with `Result`/`Option` or with `debug_assert!`.
- [ ] Add a regression test per site; add at least one `proptest` per defect class.
- [ ] Document the public API in `docs/api/v1.0.0-alpha.1.md` (intentionally unstable; do not promise stability).

**Acceptance criteria:**

- [ ] The inventory ledger shows 0 open sites of class `NaN`/`Inf` and 0 open sites of class `panic_on_valid_input`.
- [ ] `cargo test` is green on Linux x86_64, macOS aarch64, and Windows MSVC.
- [ ] `cargo fuzz` smoke run (10 min) finds no new bugs.
- [ ] A published `v1.0.0-alpha.1` artifact exists (crates.io dry-run is fine for an internal alpha).
- [ ] Internal consumers (`phenotype-bus`, `phenotype-voxel`, `phenoAI` if it depends) have acknowledged the alpha and are running on it in CI.

### 5.3 Phase 2 — Beta (≈3 weeks)

**Tag:** `v1.0.0-beta.1`

**Goal:** harden the API surface, close remaining `unwrap`/`expect` in library code, run extended fuzzing, and gather downstream feedback.

**Scope:**

- [ ] Close the remaining ~8 `unwrap`/`expect` sites (now under the `Result`/`Option` umbrella).
- [ ] Stabilise the public API: no new public items, no signature changes vs `alpha.2`, no removed items.
- [ ] Generate `api-baseline.txt` via `cargo public-api` and check it into `.audit/`.
- [ ] Run `cargo fuzz` for ≥1 hour per fuzzer target; no crashes.
- [ ] Run `cargo bench` (or criterion suite) and record baselines in `.audit/bench-baseline.txt`.
- [ ] Produce a migration guide draft (`docs/migrating-to-1.0.md`) for any pre-1.0 caller.
- [ ] Coordinate with downstream consumers to validate the API fits their use.

**Acceptance criteria:**

- [ ] Inventory ledger shows 50/50 sites closed.
- [ ] `cargo public-api` diff against the alpha tag is empty (additions allowed only behind feature flags).
- [ ] Fuzz report (1h) is crash-free and committed under `.audit/fuzz-report-beta.1.md`.
- [ ] At least two downstream crates have merged a "track phenorust 1.0-beta" PR.
- [ ] MSRV is locked; `Cargo.lock` is committed; `cargo deny` is clean.

### 5.4 Phase 3 — Release Candidate (≈2 weeks)

**Tags:** `v1.0.0-rc.1` (and `.2` only if a regression forces it)

**Goal:** produce a candidate that is feature-complete and documentation-complete, and prove it via extended fuzzing and accuracy validation.

**Scope:**

- [ ] Resolve the ~2 numerical-accuracy sites (if any) by either tightening tolerance or providing a documented "best-effort" path.
- [ ] Reach `cargo doc` zero-warning state; audit every public item for a doc comment and at least one runnable example.
- [ ] Reach `#![forbid(unsafe_code)]` (or document/audit the SIMD `unsafe` boundary).
- [ ] Run `cargo fuzz` for ≥24 hours per target; no crashes; commit report.
- [ ] Validate the migration guide against the latest pre-1.0 caller; check it into the source tree.
- [ ] Cut `v1.0.0-rc.1`; freeze the API; freeze the CHANGELOG; request sign-off from all code owners.
- [ ] Cut `v1.0.0-rc.2` *only* if a regression is found; do not add features.

**Acceptance criteria:**

- [ ] `cargo public-api` diff against the previous RC is empty.
- [ ] 24h fuzz report is crash-free and committed.
- [ ] `cargo doc --no-deps` is warning-free.
- [ ] CHANGELOG has a "1.0.0 (stable)" section in draft, listing every fixed site by ID.
- [ ] All `CODEOWNERS`-required reviewers have approved the RC.

### 5.5 Phase 4 — Stable (the 1.0 cut)

**Tag:** `v1.0.0`

**Goal:** cut the 1.0 release; publish; stabilise the worktree family for ongoing patch releases.

**Scope:**

- [ ] Cut `v1.0.0` from the RC commit. No further commits between RC and stable.
- [ ] Publish to crates.io; push the tag; open the GitHub release with the CHANGELOG excerpt.
- [ ] Promote `phenorust-wtrees/stable/` to the post-1.0 maintenance branch (`main` or `release/1.x`).
- [ ] Create `phenorust-wtrees/1.0.1-cherrypick/` for back-port discipline.
- [ ] Add a `SECURITY.md` disclosure path and a `1.0.x` patch policy (SemVer, support window).
- [ ] File follow-up issues for any deferred items (none of which may be correctness defects).

**Acceptance criteria:**

- [ ] `v1.0.0` tag exists, is signed, and points at a green CI run.
- [ ] crates.io release is live; `cargo install phenorust --version 1.0.0` succeeds.
- [ ] The Phenotype fleet's DAG (`FLEET_100TASK_DAG_v3` and successors) marks the phenorust 1.0 row as `done`.
- [ ] Patch-release branch is provisioned; `1.0.1` is the first slot, reserved for a security or correctness fix only.

## 6. Timeline Estimate

| Phase | Duration | Calendar window (illustrative) | Cumulative |
|---|---:|---|---:|
| Phase 0 — Setup | 3 days | Week 0 | 3 d |
| Phase 1 — Alpha | 3 weeks | Weeks 1–3 | ~4 w |
| Phase 2 — Beta | 3 weeks | Weeks 4–6 | ~7 w |
| Phase 3 — RC | 2 weeks | Weeks 7–8 | ~9 w |
| Phase 4 — Stable | 1 day (cut day) | Week 9 | ~9 w + 1 d |
| **Total** | **≈9 weeks** | | |

**Buffer policy:**

- A 1-week buffer is held between Alpha → Beta and Beta → RC for unplanned `unwrap`/`expect` discoveries.
- RC is capped at **2 releases** (`rc.1`, optionally `rc.2`). A third RC reopens Beta.
- Stable is cut on the next business day after a clean RC; do not delay stable to "wait for more fuzz" — the patch-release policy is the correct vehicle for post-1.0 fuzz findings.

**Assumptions driving the estimate:**

- The 50-site inventory is approximately correct (25/15/8/2 split). If the actual split skews toward `unwrap`/`expect` (which are the slowest class to fix safely), the Beta phase may need to extend by 1–2 weeks.
- Two engineers can work the inventory in parallel without merge conflict, because the sites are spread across files (typical for numerical crates). If site clustering forces serial work, add 1 week to Alpha.
- The crate has a `proptest`/fuzz harness in place; if not, add 1 week to Phase 0.

## 7. Coordination with `phenorust-wtrees/`

The fleet's worktree convention is `<repo>-wtrees/`, with children `alpha/`, `beta/`, `rc/`, `stable/` (or `<repo>-Nth`/`-wt-<task>-<date>` for ad-hoc work). For phenorust 1.0 we adopt the **named-phase** layout because the 1.0 cut is a multi-week coordinated effort.

### 7.1 Layout

```
phenorust-wtrees/
├── alpha/        # branches: release/1.0-alpha
├── beta/         # branches: release/1.0-beta
├── rc/           # branches: release/1.0-rc
├── stable/       # branches: release/1.x (post-1.0 maintenance)
└── scratch/      # ad-hoc: phenorust-wt-<task>-<YYYY-MM-DD>
```

### 7.2 Branch discipline

- `release/1.0-alpha` is cut from `main` at the start of Phase 1. Hotfixes land as fast-forward commits; once `alpha.2` is tagged, the branch is frozen and the next work happens on `release/1.0-beta`.
- `release/1.0-beta` is cut from `release/1.0-alpha` at the start of Phase 2. After `beta.1` is tagged, only `Result`/`Option` ergonomics, doc fixes, and fuzz-harness tuning are allowed.
- `release/1.0-rc` is cut from `release/1.0-beta` at the start of Phase 3. Commits allowed: doc fixes, CHANGELOG, and correctness backports. No API changes.
- `release/1.x` is cut from `release/1.0-rc` immediately after `v1.0.0` is tagged. It receives `1.0.1` cherry-picks only.

### 7.3 Coordination protocol

- The inventory ledger is the canonical work queue. Each row carries an `owner` and a `phase_target`. A worktree is provisioned per work item that exceeds one PR (named `phenorust-wt-<id>-<YYYY-MM-DD>`).
- The release captain is the single owner of the `release/1.0-*` branches. Individual contributors land fixes on `main` (or on `phenorust-wtrees/scratch/*`); the captain merges or fast-forwards into the release branch.
- The `.audit/` directory is shared across all worktrees; do not duplicate it per worktree.
- The phase tag (`alpha.1`, `beta.1`, `rc.1`, `v1.0.0`) is cut from the corresponding worktree's branch by the release captain only.

### 7.4 Downstream consumer coordination

The downstream consumers (likely `phenotype-bus`, `phenotype-voxel`, `phenoAI`, `pheno-errors`, `pheno-tracing` if they re-export numerical helpers) must:

- Pin to the latest alpha within 1 week of the alpha tag, and run their CI on it.
- File API-fit issues against `release/1.0-beta` *before* the RC tag.
- Acknowledge the RC tag within 2 business days; a no-ack counts as approval.

## 9. Risks & Mitigations

1. **Source map `phenorust-map-v2.md` cannot be recovered.**
   Mitigation: regenerate it in Phase 0 from `cargo geiger`, `cargo clippy`, and a manual `unwrap`/`expect`/floating-point audit. The 50-site count becomes a working estimate until the regenerated map is peer-reviewed.
2. **The 50-site count is wrong by ≥25%.**
   Mitigation: re-baseline at the end of Phase 0; if the count is materially higher, extend Phase 1 (Alpha) by the proportional effort and re-issue this plan as `v1.1`.
3. **A `panic!` fix exposes a deeper algorithmic defect.**
   Mitigation: triage as a Phase 2 (Beta) blocker; do not allow it to slip into RC. If it cannot be closed in Beta, defer the function behind a feature flag and document in the migration guide.
4. **Fuzz finds a crash after the RC tag.**
   Mitigation: cut `rc.2` with the fix; do not patch `rc.1`. If the crash is severe, reopen Beta and re-issue this plan.
5. **A downstream consumer cannot migrate in time.**
   Mitigation: hold the RC for at most 2 weeks; if a consumer is blocked, the migration guide must call out the breakage explicitly. Do not delay stable for one consumer.
6. **Worktree drift between `main` and `release/1.0-*` causes merge pain.**
   Mitigation: enforce fast-forward-only merges into release branches; rebase `release/1.0-*` onto `main` weekly during Alpha/Beta; stop rebasing once `rc.1` is tagged.
7. **`.audit/` directory is not tracked by git.**
   Mitigation: add `.audit/` to the repo (it is a real artifact directory, not a transient); add a `.gitignore` for raw fuzz corpora if size is a concern.

## 10. Alternative Approaches Considered

1. **Big-bang 1.0 (skip alpha/beta).**
   Trade-off: saves ~6 weeks but skips the downstream-consumer feedback loop. Rejected because the 50-site inventory is too large to bet on a single cut, and the public API is not yet stable.
2. **Time-boxed 1.0 (cut stable at week 6 regardless of site count).**
   Trade-off: forces a `1.0` that still ships defects. Rejected because correctness is the explicit 1.0 promise; a defectful `1.0.0` is more expensive than a `0.9.x` followed by `1.0.0`.
3. **Feature-flag the unsafe sites and cut 1.0 with opt-in.**
   Trade-off: lets us cut faster. Rejected because it pushes the correctness burden onto callers, which is the opposite of what 1.0 should signal.
4. **External Rust-numerics audit (third party) before RC.**
   Trade-off: adds 2–4 weeks and cost, but materially de-risks the numerical-accuracy class. **Recommended as a stretch goal** if budget permits; not on the critical path.
