# v15 Plan — 71-Pillar Cycle 4 P0 Remediation (Observability + Ops + DX + Quality)

**Theme:** Cycle-4 P0 Fleet Sweep — close the 8 lowest-scoring pillars surfaced in the
cycle-4 sweep of 5 fleet repos (Configra, pheno-config, pheno-otel, pheno-capacity,
nanovms). The single highest-leverage move is L57 (metrics RED/USE); everything else
in the cycle-4 P0 list hangs off the observability substrate.

**Date:** 2026-06-21
**Sponsor:** L5-130 (forge subagent, dispatched by orchestrator)
**Branch:** `chore/v15-plan-authoring-2026-06-21`
**Source probe:** `findings/2026-06-20-71-pillar-cycle-4-summary.md` (cycle 4 sweep)

## Goal

Close 8 P0 pillars surfaced in the cycle-4 sweep, lift the org mean for the 5 cycle-4
repos from **1.60 → 1.85+**, and ship a fleet-wide metrics RED/USE substrate
(`pheno-otel::metrics`) that future cycles can adopt without rebuilding.

## Stretch

Reach **2.00+** on the Observability domain (L57 + L58 closure), deploy the
metrics RED/USE substrate in a production cutover for at least 1 external repo
(Configra), and add `pheno-otel::metrics` to the 8-crate adoption list so v16 can
fan out without further substrate work.

## Tracks

### Track T1 — L57 Metrics RED/USE (Observability, P0 fleet-wide)

**Goal:** Ship a reusable metrics substrate in `pheno-otel::metrics` that exposes the
RED pattern (Rate, Errors, Duration) and USE pattern (Utilization, Saturation, Errors)
for any caller; adopt it in 3 of 5 cycle-4 repos (Configra, pheno-config, pheno-otel
itself).

**Why-Now:** L57 scored 1.00 across all 5 cycle-4 repos — it is the single root cause
pulling the Observability domain mean to 1.00. v14's T7 (`pheno-otel/src/metrics.rs`)
laid the groundwork but only adopted `errors.count`; full RED/USE is the next step.
Closing L57 unlocks 4 other observability-side pillars (L58, L59, L60, L61) in
future cycles without re-architecting.

**Scope:**
- Extend `pheno-otel/src/metrics.rs` with: `Counter`, `Histogram`, `UpDownCounter`
  helpers; OTLP wire format with exemplars; label cardinality guardrails.
- Add 3 metric emit points to Configra (HTTP request handler), pheno-config (loader
  reload), pheno-otel (gRPC stream lifecycle).
- Update `pheno-otel/README.md` with a RED/USE cookbook.

**Acceptance:**
- [ ] `pheno-otel::metrics` exports `counter!`, `histogram!`, `gauge!` macros with
  OTLP serialization (test: round-trip via `MockExporter`)
- [ ] 3 repos emit at least 1 RED metric + 1 USE metric per request path (verified by
  integration test capturing exported metrics)
- [ ] Label cardinality guardrail (test: 10k distinct label values rejected at runtime)
- [ ] Cookbook in `pheno-otel/README.md` shows 3 worked examples (HTTP server, gRPC
  client, batch job)

**Effort:** M (3-4h on MacBook; no external resources)

**Risks:**
- R1: Cardinality explosion in fleet — mitigated by guardrail + cardinality dashboard
- R2: OTLP wire format drift between OTel SDK versions — pinned via
  `pheno-otel/Cargo.toml` `opentelemetry = "0.27"` + integration test

**Depends-on:** v14 T7 (`pheno-otel::metrics` skeleton) — **already merged**.

**Owner:** parallel-agent (TBD by orchestrator dispatch)

### Track T2 — L58 Distributed Tracing (Observability, P0 fleet-wide)

**Goal:** Wire W3C `traceparent`/`tracestate` end-to-end across the 5 cycle-4 repos so
a single trace-id flows from Configra HTTP ingress through pheno-config loader →
pheno-otel exporter → pheno-capacity backplane.

**Why-Now:** L58 scored 1 across all 5 cycle-4 repos; without trace continuity,
T1's RED metrics cannot be correlated across hops. v12 T3 added W3C context in
pheno-otel alone; cross-tier propagation is the gap.

**Scope:**
- Add `traceparent` header injection in Configra's HTTP middleware
  (tower-http `TraceLayer`).
- Add `traceparent` propagation in pheno-config's loader boundary (incoming
  `LoaderContext` carries `TraceContext`).
- Verify propagation through pheno-otel (existing trace-context test suite from v12 T3)
  via 1 new integration test that asserts trace-id continuity.
- Document the propagation contract in `docs/observability/trace-context.md`.

**Acceptance:**
- [ ] Configra emits `traceparent` on all outgoing HTTP requests (test: 2-hop trace
  continuity from synthetic client)
- [ ] pheno-config's loader surfaces `TraceContext` and propagates it (test:
  LoaderContext carries span context)
- [ ] End-to-end test asserting single trace-id spans Configra → pheno-config →
  pheno-otel → pheno-capacity
- [ ] `docs/observability/trace-context.md` documents the propagation contract with
  sequence diagram

**Effort:** M (2-3h on MacBook)

**Risks:**
- R1: HTTP client library swap (reqwest → hyper) — mitigated by `TraceLayer` adapter
- R2: gRPC streaming context propagation — already covered by pheno-otel test suite

**Depends-on:** v12 T3 (W3C trace-context test suite) — **already merged**.

**Owner:** parallel-agent (TBD by orchestrator dispatch)

**Risks:**

### Track T4 — L62 Backup/Restore (Ops, 4 of 5 cycle-4 repos)

**Goal:** Ship a `pheno-backup` CLI binary + adopt it in Configra, pheno-config,
pheno-capacity (3 of 5 cycle-4 repos; nanovms and pheno-otel are stateless or
config-only and out of scope).

**Why-Now:** L62 scored 1 across 4 of 5 cycle-4 repos (nanovms exempt: stateless).
Without backup/restore, the fleet cannot recover from operator error or
data-corruption events.

**Scope:**
- Author `pheno-backup` Cargo binary (~200 lines) with `backup` and `restore`
  subcommands; supports S3 (`aws-sdk-s3`) and local-filesystem targets.
- Integrate into Configra (state DB), pheno-config (config snapshot dir),
  pheno-capacity (capacity ledger).
- Document runbook in `docs/ops/backup-restore.md`.

**Acceptance:**
- [ ] `pheno-backup backup --target <s3|fs> --source <path>` produces a tarball +
  manifest JSON
- [ ] `pheno-backup restore --manifest <path>` restores from backup with checksum
  verification
- [ ] 3 repos (Configra, pheno-config, pheno-capacity) invoke `pheno-backup` in their
  nightly CI job (cron-style)
- [ ] Runbook covers: backup cadence, retention policy (30d), restore drill
  procedure, integrity-check command

**Effort:** L (5-6h on MacBook; L because of S3 SDK integration)

**Risks:**
- R1: aws-sdk-s3 credential chain complexity — mitigated by env-var-only auth + runbook
- R2: Restore-from-backup race conditions — mitigated by `flock`-based lock file

**Depends-on:** none (greenfield CLI)

**Owner:** parallel-agent (TBD by orchestrator dispatch)

### Track T5 — L63 Capacity Planning (Ops, all 5 cycle-4 repos)

**Goal:** Adopt `pheno-capacity` (existing CLI) into the 5 cycle-4 repos' CI gates:
fail CI if projected resource utilization exceeds 80% at the planning horizon.

**Why-Now:** L63 scored 1 across all 5 cycle-4 repos; without capacity gates, fleet
growth hits resource exhaustion before ops can react. `pheno-capacity` is built
(L5-122 bucket-table drift resolved); adoption is the gap.

**Scope:**
- Add `capacity-gate` CI job to Configra, pheno-config, pheno-otel, pheno-capacity
  (self), and nanovms.
- Each job runs `pheno-capacity plan --horizon 30d`; fails if any resource category
  projects >80% utilization.
- Document the gate semantics in `docs/ops/capacity-gate.md`.

**Acceptance:**
- [ ] 5 CI workflows invoke `pheno-capacity plan` and fail on >80% projection
- [ ] Test scenario: artificially inflate projected load → CI fails with
  `capacity-gate failed: cpu 92% > 80% threshold`
- [ ] Docs cover: threshold tuning, override procedure, integration with
  `pheno-backup` (T4) for state snapshots

**Effort:** S (1.5h on MacBook; ~30 lines of YAML per repo)

**Risks:**
- R1: pheno-capacity false positives during load tests — mitigated by tag-based
  override (`capacity:skip-gate`)
- R2: Threshold drift across repos — single-source `pheno-capacity.toml` per repo,
  loaded at runtime

**Depends-on:** L5-122 (bucket-table drift resolution) — **already merged**;
pheno-capacity binary — **already built**.

**Owner:** parallel-agent (TBD by orchestrator dispatch)

## Cross-Cutting Concerns

1. **CI billing.** Per the GitHub Actions billing policy (AGENTS.md §24), all 8
   tracks must use standard Linux runners only. macOS runners are banned. The
   chaos suite (T8) uses weekly scheduling to stay within the budget; the
   perf-gate (T3) and capacity-gate (T5) are per-PR but lightweight (single-job).

2. **Drift between parallel-agent branches.** With 8 concurrent PRs touching 5
   cycle-4 repos, cross-PR conflicts are likely on shared CI workflows and
   `audit_scorecard.json`. Mitigation:
   - Each track writes to its own `scorecard.<track>.json` and a post-merge
     aggregator reconciles into `audit_scorecard.json`.
   - Shared CI workflows (e.g., pheno-otel's `ci.yml`) are touched by T1, T2,
     T3, T5, T8 — track owners must coordinate via the orchestrator's
     `conflict-watcher` subagent.

3. **Branch-naming hygiene.** The branch-naming check (v14 side-51) accepts
   `v15-T<n>-<short-desc>-2026-06-21` pattern. All 8 tracks must follow this
   convention; the orchestrator's PR-bot rejects non-conforming branches.

4. **Conventional Commits.** All 8 tracks must use Conventional Commits
   prefixes: `feat(pheno-otel)` for T1+T2, `feat(pheno-config)` for T6, etc.
   The commitlint hook (v14 side-50) enforces.

5. **Worklog schema.** All 8 tracks produce v2.1 worklog JSON per ADR-025:
   `worklogs/L5-13X-v15-T<n>-<short-desc>-2026-06-21.json`. Each worklog
   declares `plan_ref: plans/2026-06-21-v15-71-pillar-cycle-4-p0.md` and
   `pillar_ref: L<NN> (<pillar name>)` for traceability.

6. **Test-data integrity.** T1, T2, T3, T5, T8 each introduce test fixtures
   (metric captures, trace dumps, bench baselines, capacity projections, chaos
   traces). Fixtures must be deterministic and committed under
   `tests/fixtures/<track>/` so future cycles can re-run.

7. **External-service credentials.** T4 (`pheno-backup` with S3 target) requires
   `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY` for the test bucket. The
   test bucket is `phenotype-backup-test` (us-west-2). Credentials are
   injected via `secrets.PHENO_BACKUP_AWS_*` and only mounted for the T4
   CI job.

8. **Rollback strategy.** Each track's PR is independently revertable. If T1
   breaks the metrics substrate, T2's adoption can be reverted independently
   (T2 has a feature-flag `PHENO_OTEL_METRICS_V2`). Cycle-5 probe re-runs the
   71-pillar framework and confirms no regressions in closed pillars.
