## T1 — Grafonnet/Jsonnet pipeline

**Goal:** bring the 5 Grafana dashboards (L25 v22-T1) under code review
by expressing them as Grafonnet/Jsonnet modules committed to git.

**Deliverables:**
- `dashboards/grafonnet/fleet-traffic.libsonnet` (golden-signal: traffic)
- `dashboards/grafonnet/per-adapter-latency.libsonnet` (latency)
- `dashboards/grafonnet/cache-saturation.libsonnet` (saturation)
- `dashboards/grafonnet/connection-lifecycle.libsonnet` (errors)
- `dashboards/grafonnet/otel-export-health.libsonnet` (data health)
- `dashboards/Makefile` — `make render` runs `jsonnet -J` to produce
  dashboard JSON; `make deploy` posts to Grafana
- `dashboards/README.md` — usage, code review process
- `.github/workflows/dashboards-ci.yml` — render + diff guard

**Pillar impact:** L25 2.5 → 3.0 (dashboards now code-reviewed)

**Wall:** 1 week (5 days)
**LOC:** 200
**Owner:** orchestrator (single repo, no sub)

## T2 — OTel collector HA (CRITICAL PATH)

**Goal:** run the OTel collector with 2+ replicas behind a load
balancer, so loss of a single collector does not lose spans.

**Deliverables:**
- `infra/otel-collector/deployment.yaml` (k8s Deployment w/ 2 replicas)
- `infra/otel-collector/service.yaml` (ClusterIP)
- `infra/otel-collector/ingress.yaml` (gRPC + HTTP LB)
- `infra/otel-collector/config-ha.yaml` (multi-receiver, per-replica tag)
- `infra/terraform/otel-collector.tf` (provisions a 2-replica deployment
  on the prod cluster)
- `docs/runbooks/otel-collector-ha.md` (incident playbook: replica
  failover, span buffer overflow, exporter throttle)
- `scripts/otel-collector/healthcheck.sh` (liveness probe + 5xx budget)

**Pillar impact:** L59 2.5 → 3.0 (HA infra deployed)

**Wall:** 1 week (5 days)
**LOC:** 400
**Owner:** orchestrator + subagent (sub owns the terraform)
**Subagent:** heavy-runner for terraform plan + apply to staging

## T3 — SLO burn-rate alerts (CRITICAL PATH)

**Goal:** define SLOs for the 5 golden signals, then add burn-rate
alerts so we page on SLO violations before users notice.

**Deliverables:**
- `docs/slo/traffic-slo.md` (request rate baseline + growth model)
- `docs/slo/latency-slo.md` (p99 budget: 200ms @ 100rps)
- `docs/slo/saturation-slo.md` (cache hit rate floor: 80%)
- `docs/slo/errors-slo.md` (error rate ceiling: 0.1%)
- `docs/slo/data-slo.md` (OTel drop rate ceiling: 0.5%)
- `infra/grafana/alerts/slo-burn-rate-fast.yaml` (1h window, 2% budget)
- `infra/grafana/alerts/slo-burn-rate-slow.yaml` (6h window, 5% budget)
- `docs/runbooks/slo-burn-rate-response.md` (pager playbook)

**Pillar impact:**
- L63 2.0 → 2.5 (burn-rate alerts live)
- L14 2.5 → 3.0 (SLOs now define the p99 budget)

**Wall:** 1 week (5 days)
**LOC:** 300
**Owner:** orchestrator (single repo, no sub)

## T4 — Adaptive sampling

**Goal:** replace the current "sample 100% of traces" with head-based
adaptive sampling (sample more for slow/errored traces, less for
healthy) so we can sustain 10k rps without OTel cost explosion.

**Deliverables:**
- `pheno-tracing/src/sampling.rs` (tail-based sampler w/ rules)
- `pheno-tracing/src/rules.rs` (rule DSL: per-endpoint, per-status, per-latency)
- `pheno-tracing/tests/sampling_proptest.rs` (proptest coverage of
  rule combinations)
- `pheno-tracing/examples/sampling-rule.toml` (default rule)
- `docs/observability/sampling-guide.md` (when to add a rule)

**Pillar impact:** L26 2.5 → 3.0 (coverage gate now considers sampling
correctness)

**Wall:** 4 days
**LOC:** 150
**Owner:** subagent (macbook-friendly, no cargo contention)

## T5 — K8s operator (CRITICAL PATH)

**Goal:** ship a minimal k8s operator that watches a CRD
`PhenoApp` and reconciles to a Deployment + Service + Ingress.

**Deliverables:**
- `phenokube/crds/phenoapp.yaml` (CRD: replicas, image, ports, env)
- `phenokube/src/operator.rs` (controller loop)
- `phenokube/src/reconcile.rs` (Deployment + Service + Ingress)
- `phenokube/src/crd.rs` (CRD type + validation)
- `phenokube/Cargo.toml` (k8s-openapi + tokio + thiserror)
- `phenokube/README.md` (CRD spec, install, test)
- `phenokube/tests/integration.rs` (test against kind cluster)
- `.github/workflows/phenokube-ci.yml` (build + kind test)
- `docs/operators/phenoapp.md` (CRD + usage)

**Pillar impact:**
- L33 2.0 → 2.5 (dev surface now has k8s operator)
- L36 2.5 → 3.0 (chaos-in-CI can now use k8s operator to inject failures)
- L37 2.5 → 3.0 (chaos-in-CI weekly can leverage operator for fleet)

**Wall:** 1 week (5 days)
**LOC:** 500
**Owner:** subagent (heavy-runner, needs k8s for testing)
