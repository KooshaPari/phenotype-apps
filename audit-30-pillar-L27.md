# L27 — Observability of failure modes (SLO/SLA / error budget / burn rate / runbooks / postmortem)

## Scope
Catalogue every failure-observability artifact: SLO/SLA definitions, error budget tracking, burn rate alerts, SLO dashboards, runbooks, postmortem templates, incident playbooks. Both in-repo and across the wider Phenotype bloc.

## SOTA 2026
- **Google SRE Workbook (Beyer, Jones, Petoff, Murphy, 2017)** — SLI/SLO/burn-rate, multi-window multi-burn-rate alerting (MWMB)
- **Jones et al., "Service Level Objectives in Practice" (2019)** — burn-rate algebra, error budget consumption
- **Atlassian Incident Handbook, Blameless Postmortems (Kerr, 2018)**
- **PagerDuty Incident Response (Stine, 2014-2024 editions)** — severity levels, escalation matrix
- **NIST SP 800-61r2 (Cichonski et al., 2012) / SP 800-184 (Grance et al., 2016)** — incident handling/recovery
- **OpenSLO / Sloth / pyrra** — declarative SLO-as-code + Prometheus rule generation

2. **Add `/metrics` (Prometheus) + `/healthz` + `/readyz` + `/startupz` axum routes** to `agileplus-api` modeled on Tracera's `ProbeType` triad.
3. **Create `docs/RUNBOOK.md` and `docs/POSTMORTEM_TEMPLATE.md` in AgilePlus and Tracely** — port thegent's runbook structure; create a blameless postmortem template (summary, timeline, root cause, contributing factors, action items, lessons).

5. **Create `.github/workflows/chaos-tests.yml`** in Tracera (referenced but missing) and AgilePlus, running on weekly cron.
6. **Populate `Tracera/tests/chaos/`** with chaos-toolkit experiment definitions; wire to chaos-tests workflow.
7. **Add SLO contracts to AgilePlus SPEC.md** for every HTTP route — concrete `SloRegulator` thresholds per endpoint matching the benchmark gates.
8. **Build a Grafana SLO dashboard** with golden-signal panels (latency, traffic, errors, saturation) per service, sourced from Prometheus exporter.
9. **Adopt OpenSLO** YAML manifests under `AgilePlus/slo/` (one file per service) — version-controlled SLO-as-code.
10. **Wire thegent's `IncidentSnapshotBundle`** into the existing `service_failure.rs` chaos test — every chaos run should produce a snapshot for retrospective analysis.
