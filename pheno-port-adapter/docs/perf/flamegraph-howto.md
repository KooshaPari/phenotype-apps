# Reading the pheno-port-adapter flamegraph

This document explains how to **read** the `flamegraph.svg` produced by the L44 bench (`benches/flame.rs`), what each axis means, and what "good" vs "bad" looks like for the 4-stage / 5-component port-adapter pipeline.

## 1. What is in this SVG

The flamegraph captures one run of the full pipeline:

```
source bytes
     │
     ▼
┌─────────┐    ┌──────────┐    ┌────────┐    ┌──────────┐    ┌─────────┐
│ Scanner │ ─► │ Analyzer │ ─► │ Mapper │ ─► │ Compiler │ ─► │ Emitter │
└─────────┘    └──────────┘    └────────┘    └──────────┘    └─────────┘
byte→token   token→AST     AST→IR    IR→dispatch    IR→bytes
```

The five functions you should be able to find in the SVG, named exactly:

- `pheno_port_adapter::benches::flame::scanner`
- `pheno_port_adapter::benches::flame::analyzer`
- `pheno_port_adapter::benches::flame::mapper`
- `pheno_port_adapter::benches::flame::compiler`
- `pheno_port_adapter::benches::flame::emitter`

If any of these names is missing from the SVG, the optimizer inlined or dead-code-eliminated the stage — bump the `#[inline(never)]` annotation on `run_pipeline_final_only` (already on) and the `cargo build --benches --release` invocation must include `debug = true` in `[profile.bench]` so debug info survives LTO.

## 2. Axes

A flamegraph is read **upside-down** compared to most charts:

| Axis | Meaning |
|------|---------|
| **Y (vertical)** | Stack depth. The root is on the bottom; the leaf that consumed the most CPU is on top. Wider towers = more time spent in that frame's subtree. |
| **X (horizontal)** | Total CPU time (sample count). Each column is one stack, sorted alphabetically (NOT chronologically) so wide regions are easy to spot. |
| **Color** | Decorative in the SVG generated by `pprof::flamegraph::Options` — colors are picked from a hash of the function name. Use color to group visually, not to encode meaning. |
| **Title** | "pheno-port-adapter / 4-stage pipeline (Scanner->Analyzer->Mapper->Compiler->Emitter)" — this is set from `Options::title` in `benches/flame.rs`. |
| **Subtitle** | "L44 flamegraph -- N s profile, 100 Hz sampling" — set from `Options::subtitle`. |

## 3. Reading the five stages

Each of the five components is a `fn` you can click into the SVG. The expected width share for a healthy run on a 10 KiB synthetic source (~600 entries):

| Stage | Expected share | What it would mean if it's much wider |
|------|----------------|--------------------------------------|
| `scanner` | 25–40% | Byte loop is the hot spot — consider `aho-corasick` or `memchr::memchr` for keyword scanning, or a hand-written DFA. |
| `analyzer` | 10–20% | Token walk is expensive — the `let-else` inside the `if let Tok::Ident(...)` chain may be the cost; consider a `match` with explicit byte offsets. |
| `mapper` | 5–15% | HashMap churn dominates — switch the `BTreeMap` to `IndexMap` if insertion order matters, or shard the `RouteTable` by kind. |
| `compiler` | 5–15% | The fnv1a + std-hash XOR fold is doing real work — fold it into the `DefaultHasher` only if you need determinism; otherwise drop the fnv pass. |
| `emitter` | 5–15% | The `Vec<u8>` growth is the cost — pre-size with `Vec::with_capacity(total_estimated_bytes)`, or switch to `bytes::BytesMut`. |

The remaining share (20–40%) will be spent in `std::*`, `alloc::*`, and `core::*` — particularly `alloc::raw_vec`, `core::fmt::write`, and `std::collections::hash::map::DefaultHasher::write`.

## 4. Reading common frames

A few non-obvious frames you'll see and what they mean:

- **`alloc::raw_vec::RawVec<T,A>::reserve_for_push`** under `emitter` → the `Vec<u8>` grew past its capacity more than once. Fix: pre-size.
- **`std::str::from_utf8`** under `scanner` → the scanner is copying byte slices into a `String` per ident/string. Acceptable; if it dominates, switch to `Cow<str>` or `&str` slices into the source.
- **`std::collections::hash::map::DefaultHasher::write`** under `compiler` → the SipHash mixing is paying the rent for the stable hash. Acceptable.
- **`<pprof::ProfilerGuard as core::ops::drop>::drop`** at the top of the SVG → the guard is taking ~1 ms to flush samples. This is normal; it shows up as a thin tall bar because the stack is shallow.
- **`pprof::flamegraph::…`** near the SVG-generation side → that's the SVG serializer and should NOT appear in the timed region. If you see it inside `run_pipeline_final_only`, the `Once` guard in `maybe_capture_flamegraph` is broken — the SVG write is happening inside the iteration loop.

## 5. Diffing two flamegraphs

When you want to compare "before" vs "after" of a change:

1. Capture the baseline: `cargo bench --bench flame -- --profile-time=10 --save-baseline main` then `cp flamegraph.svg flamegraph.main.svg`. (Note: the `Once` capture happens on the FIRST invocation only, so rerunning the bench with `--baseline` won't re-capture. If you need a fresh capture, delete `flamegraph.svg` first.)
2. Make the change.
3. Capture the candidate: `PHENO_FLAMEGRAPH_OUT=flamegraph.after.svg cargo bench --bench flame -- --profile-time=10`.
4. Open both in your browser side-by-side and look for **new towers** or towers that grew by > 10%.

The criterion harness also writes a `target/criterion/port_pipeline_4stage/` HTML report with statistical comparisons if you keep the bench output around.

## 6. Common pitfalls

- **"The flamegraph is empty / only has `_start` and `__libc_start_main`."** — The bench ran but the optimization elided everything. Make sure `[profile.bench]` has `debug = true` and that `run_pipeline_final_only` has `#[inline(never)]`.
- **"I see `pheno_port_adapter::benches::flame::run_pipeline_final_only` but no Scanner/Analyzer/Mapper/Compiler/Emitter."** — The optimizer inlined the stages. Same fix as above; or use `#[inline(never)]` on each stage.
- **"The SVG is 50 MB and Chrome chokes."** — The `pprof` sampling rate is too high. Lower from `100` Hz to `99` or fewer in `pprof::ProfilerGuard::new(...)`. 100 Hz is the default for a 10 s profile and is fine; only reduce if you go above 30 s.
- **"My flamegraph shows 90% `std::collections::hash::map::DefaultHasher::write`."** — That's intentional if your change moved work into the hash stage; otherwise check that you didn't accidentally allocate a `HashMap` per iteration.
- **"I'm on macOS and `cargo bench` says `failed to read profile data`."** — pprof falls back to a software sampler on macOS; the SVG is still produced. If it's empty, increase the `seconds` parameter to ≥ 5 s so the sampler has enough signal.

## 7. Where the SVG lives

| Path | What it is |
|------|-----------|
| `flamegraph.svg` (crate root) | Default location for the local bench invocation. |
| `PHENO_FLAMEGRAPH_OUT=<path>` env var | Override the output path. The CI workflow uses this to point at the absolute path it uploads. |
| GitHub Actions artifact `pheno-port-adapter-flamegraph` | The CI-uploaded SVG, available from every PR build. |

## 8. Reproducing the SVG locally

```bash
# 1. Build (release, with debug info for symbols)
cargo build --benches --release

# 2. Run the bench — the flamegraph.svg is dropped in the crate root
cargo bench --bench flame -- --profile-time=10

# 3. Inspect
open flamegraph.svg        # macOS
xdg-open flamegraph.svg    # Linux
```

If you want a longer or shorter profile window without touching the bench code, set `PHENO_FLAME_PROFILE_SECS`:

```bash
PHENO_FLAME_PROFILE_SECS=30 cargo bench --bench flame -- --profile-time=10
```

The first env var controls the **flamegraph capture window** (30 s in the example above); the second (`--profile-time=10`) controls the **criterion measurement window**. They are intentionally separate so CI can keep the criterion measurement at 10 s for stability while letting engineers capture deeper profiles locally.
